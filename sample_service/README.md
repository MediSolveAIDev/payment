# Sample Shop — 구독결제 연동 샘플 (실제 토스 연동, 카드 보관함 모델)

payment_system 구독서버의 **외부 서비스 역할을 하는 Django 참고 구현**이다.
실제 토스페이먼츠 테스트 키로 카드 등록 → 빌링키 발급 → 결제 승인까지 실제 API로 동작하며,
**서비스 개발팀이 화면을 눌러 보면서 연동 방법을 배우고 코드를 그대로 가져다 쓰도록** 만들어졌다.

## 카드 보관함(Card Vault) 모델 — 중요 변경

구독서버가 **카드 보관함 모델**로 전환되어, 카드 등록이 구독·결제와 **분리**되었다.

- **카드 등록/변경**: 토스 빌링 인증(authKey) → `POST /api/v1/cards`.
  같은 사용자로 **다시 등록하면 기존 카드가 교체**된다 — 이것이 곧 "카드 변경"이다(별도 change-card API 없음).
- **구독 생성 / 단건 결제**: `POST /api/v1/subscriptions`, `POST /api/v1/payments` 호출 시
  **토스 authKey/customerKey가 더 이상 필요 없다.** 서버가 사전 등록된 카드의 빌링키로 결제한다.
  카드가 없으면 서버가 `404`를 반환하므로, 먼저 카드를 등록해야 한다.
- 카드 화면(`/card`)이 카드 보관함의 단일 진입점이다 — 조회(`GET /api/v1/cards/{uid}`),
  등록/변경(`POST`), 삭제(`DELETE`). 사용 중인 구독이 있는 카드는 삭제 시 `409`.

> 권장 데모 순서: **이메일 선택(`/login`) → 서비스 선택(`/services`) → 카드 등록(`/card`) → 요금제 구독(`/plans`) 또는 일반 결제(`/pay`)**.
> 구독/결제 화면에서 카드가 없으면 자동으로 `/card?next=…`로 유도하고, 등록을 마치면 원래 흐름으로 복귀한다.

## 화면이 곧 문서 — 「개발자 노트」

모든 화면 하단에 다크 패널 **「🔍 개발자 노트」** 가 있다. 펼치면 그 화면이:

- **어떤 API를 호출하는지** (메서드·경로·대응 `payment_client.py` 함수)
- **어느 파일·함수가 구현하는지** (`shop/views.py`의 뷰 함수명)
- **꼭 따라야 할 규칙** (금액 변조 방지, 타임아웃≠실패, 멱등 order_id 등)

을 그 자리에서 보여준다. 화면 흐름(이메일 선택 → 서비스 선택 → 카드 등록 → 구독/결제)을
한 번 따라가면 연동의 전 과정을 이해할 수 있다.

## 내 서비스에 가져다 쓰는 법

| 가져갈 것 | 파일 | 설명 |
|---|---|---|
| **API 클라이언트 (핵심)** | `shop/payment_client.py` | HMAC 3중 인증 서명 + 카드 보관함 포함 전체 API 호출. **이 파일 하나만 복사하면 연동 끝** — Django 의존 없음(`settings` 폴백 2줄만 교체) |
| 화면 흐름 참고 | `shop/views.py` | 카드 등록 successUrl 처리(`POST /api/v1/cards`), 카드 미등록 시 `/card` 유도, 401 키 재입력 흐름 등 패턴 |
| 토스 SDK 호출부 | `shop/templates/shop/card.html` | 카드 등록 위젯 `requestBillingAuth()` 호출과 successUrl(`/billing/success?next=…`) 구성. 토스 키 없이 테스트할 수 있는 **수동 authKey 폴백** 포함 |
| 디자인(선택) | `shop/templates/shop/base.html` | Centurion 디자인 토큰 + 컴포넌트가 한 파일에 — 외부 CSS 의존 없음 |

연동 시 지켜야 할 핵심 규칙(개발자 노트에도 동일하게 표기):

1. **타임아웃(503 PAYMENT_UNRESOLVED)은 실패가 아니다** — 서버가 PENDING 유지 후 정산 스윕으로 확정한다. 실패 처리하지 말 것.
2. **금액은 서버 세션/DB에 보관**했다가 서명된 본문으로 전달 — successUrl 쿼리를 신뢰하지 말 것.
3. **order_id는 서비스 내 고유 + 멱등** — 같은 order_id 재요청은 기존 결제가 반환된다(이중결제 방지).
4. 접근 제어는 구독 조회 응답의 `access_allowed` 하나로 판단 — 상태별 분기를 직접 구현하지 말 것.

## 셋업

1. **구독서버에서 서비스 등록** (http://127.0.0.1:8000/admin)
   - 서비스 관리 > 서비스 등록 — 허용 IP에 `127.0.0.1`
   - 등록 직후 키 화면(또는 서비스 상세의 **키 복사** 버튼)에서 API 키/HMAC Secret 복사
   - 요금제 1개 이상 생성 (체험 요금제 포함 권장)
2. **이 프로젝트 설정**
   ```bash
   cp .env.example .env   # SERVICE_API_KEY / SERVICE_HMAC_SECRET 채우기
   python3 -m venv .venv && .venv/bin/pip install -r requirements.txt
   .venv/bin/python manage.py migrate
   ```
3. **두 서버 구동**
   ```bash
   # 터미널 1 — 구독서버 (payment_system)
   cd ../payment_system && .venv/bin/uvicorn app.main:app --port 8000
   # 터미널 2 — 샘플 서비스
   .venv/bin/python manage.py runserver 8001
   ```

## 실결제 테스트 시나리오 (수동) — 카드 보관함 흐름

1. http://127.0.0.1:8001 접속 → **이메일 입력**(1단계, `/login`) → **서비스 선택**(2단계, `/services`)
2. **카드 등록**: 상단 메뉴 **카드**(`/card`) → "카드 등록창 열기" → 실제 토스 카드등록창에서
   테스트 카드 입력(토스 테스트 모드에서는 청구되지 않음). 완료 시 `POST /api/v1/cards`로
   빌링키가 발급·보관되고 마스킹 카드 정보가 표시된다.
   - 토스 client key가 없으면 카드 화면의 **"수동 authKey 입력"** 폴백으로 등록 흐름을 테스트할 수 있다.
3. **구독**: `/plans`에서 요금제 선택 → "구독하기" → 확인 화면에서 **등록 카드로 즉시 구독**
   (토스 인증 재진입 없음). 카드가 없으면 자동으로 카드 등록(`/card`)으로 유도된 뒤 복귀한다.
4. 확인 지점:
   - 샘플: `/my`에서 상태 ACTIVE, 카드/다음 결제일
   - 구독서버 admin: 구독·결제 내역
   - [토스 개발자센터](https://developers.tosspayments.com/) 테스트 결제내역
5. **카드 변경**: `/card`(또는 `/my`의 "카드 변경")에서 카드 등록창을 다시 열어 재등록.
   `POST /api/v1/cards` 재호출로 기존 카드가 교체되며, **기존 구독이 새 카드를 자동 참조**한다
   (별도 구독 API 불필요).
6. 라이프사이클: `/my`에서 **구독 취소**(만료일까지 유지) → **재개**, SUSPENDED/PAST_DUE 시 **수동 결제**
   - **PAST_DUE(연체) 수동 결제**: 자동 재시도가 반복 실패하면 상태가 PAST_DUE로 전환.
     `/my` 화면에 **취소 / 카드 변경 / 수동 결제** 버튼이 함께 노출되어 즉시 수동 결제가 가능.
     PAST_DUE는 아직 접근이 허용된 상태이므로 구독 카드 위에 "연체 경고" 배너(⚠️)가 표시된다.
7. 체험: 체험 요금제에서 "무료 체험" — 결제 없이 TRIAL 시작, 만료 시 등록 카드로 자동 결제
   (체험도 사전 카드 등록 필요)

### 결제 내역 (/history)

상단 메뉴 **결제 내역**(`/history`) 또는 직접 접속:

- **구독 결제 내역**: `GET /api/v1/payments/{uid}` 를 호출해 구독서버로부터 받아온 내역.
  서버가 Subscription join으로 필터하므로 **구독 결제만** 반환된다 (단건 미포함).
- **단건(일반) 결제 내역**: 이 샘플 서비스가 로컬 DB `OneOffRecord` 에 저장한 기록.
  두 목록을 하나의 화면에서 통합 조회한다.
- **단건 취소**: 단건 내역의 미취소 건에는 **취소** 버튼이 노출되며,
  `POST /pay/cancel` → 구독서버 `POST /api/v1/payments/{order_id}/cancel` 로 전달된다.
  취소 성공 시 로컬 `OneOffRecord.canceled = True` 로 업데이트하고 `/history` 로 돌아온다.
  서비스 취소 정책(`cancellation_enabled`)이 꺼져 있으면 `CANCEL_DISABLED` 에러가 표시된다.

### 받은 알림 (/notifications) — 서비스 알림 수신 데모(요청 016)

구독서버는 구독·결제·카드·요금제 이벤트가 발생하면 서비스가 등록한 **알림 URL**로 JSON을 POST한다.
- **수신 엔드포인트**: `POST /notify` — HMAC 서명(`X-Signature`/`X-Timestamp`/`X-Nonce`)을 검증해 `NotificationRecord`에 저장(`shop/views.py:notify_receive_view`). 서명 방식은 API 호출 서명과 동일.
- **화면**: 상단 메뉴 **받은 알림**(`/notifications`)에서 수신 알림 목록(EVENT·상태·사용자·설명·서명검증)을 본다. 화면 상단에 **등록용 수신 URL(복사 버튼)**이 표시된다.
- **등록**: 이 샘플은 결제 서버와 **별도 docker**이므로 결제 서버 컨테이너에서 닿으려면 **`http://host.docker.internal:8001/notify`**를 등록한다(`localhost:8001`·`sample:8000`은 닿지 않음). `/notifications` 화면 상단에 이 주소가 복사 버튼과 함께 표시된다. 등록 후 어드민의 **‘테스트 알림 전송’** 버튼으로 연결을 확인할 수 있다.
- `config/settings.py`의 `ALLOWED_HOSTS`에 `host.docker.internal`을 포함해야 Django가 알림 POST를 받는다(없으면 400 DisallowedHost).

### 일반(단건) 결제 — 구독과 무관한 1회성 결제

상단 메뉴 **일반 결제**(`/pay`) 또는 직접 접속:

1. 상품명 + 금액 입력 → "결제 진행"
2. 서버가 **등록된 카드(카드 보관함)의 빌링키로 즉시 1회 결제**(`POST /api/v1/payments`).
   토스 인증 재진입 없음 — 카드가 없으면 카드 등록(`/card?next=/pay`)으로 유도된다.
3. 확인 지점:
   - 샘플: 결과 화면에 주문번호·금액·상태(DONE)
   - 구독서버 admin: **결제** 메뉴에서 종류 "일반"으로 표시 / **정산**에서 일반 금액으로 집계 /
     대시보드 "이번달 일반결제" 카드
   - 구독과의 차이: 구독 슬롯/만료가 없음(카드는 보관함에 그대로 유지되어 재결제에 재사용)

> 금액은 이 서비스가 정한 값이며 결제 시 HMAC 서명된 본문으로 전달된다(중간자가 amount를 변조하면
> 서명 검증에서 거부). 주문번호(order_id)는 `oo-<uuid>`로 자동 생성 — 같은
> order_id 재요청은 구독서버가 멱등 처리(이중결제 방지).

## 다중 서비스 테스트 흐름

구독서버에 여러 서비스를 등록한 경우, 이 샘플 앱으로 **각 서비스를 전환하며 전체 테스트**가 가능하다.

### 1. 서비스 목록 화면 (`/services`)

이메일 선택(`/login`) 직후(또는 언제든 상단 내비 **서비스** 클릭) `/services`로 이동한다.
로그인 없이 `/services`에 접근하면 자동으로 `/login`으로 유도된다.

- 화면 로드 시 구독서버 `GET /api/v1/services`를 **인증 없이** 호출해 서버에 등록된 서비스 목록(id·이름·상태)을 가져온다.
- 각 서비스 행에 **"키 저장됨"** 배지가 있으면 `[선택]` 버튼으로 즉시 활성화(키 재입력 불필요).
- 키가 저장되지 않은 서비스에는 **api_key·hmac_secret 입력 폼**이 표시된다. 어드민 서비스 생성 시 **1회 발급된 평문 키**를 붙여 넣고 "저장 후 선택"을 누르면 DB에 저장되어 이후에는 묻지 않는다.

### 2. 키 저장 규칙

| 상황 | 동작 |
|---|---|
| 저장된 키 있음 | `[선택]` 버튼만 노출 — 키 재입력 불필요 |
| 저장된 키 없음 | api_key·hmac_secret 입력 폼 표시 |
| 키 저장 완료 | DB에 보존(`ServiceCredential`) — 앱 재시작 후에도 유지 |
| .env 단일 키 | 활성 서비스가 없을 때 폴백(단일 서비스 환경 하위호환) |

### 3. 활성 서비스 표시 및 전환

- 활성 서비스가 설정되면 **상단 내비**에 현재 서비스명이 표시된다(`user@example.com · 서비스A`).
- **서비스 변경** 링크를 클릭하면 언제든 `/services`로 이동해 다른 서비스로 전환할 수 있다.
- 전환 즉시 요금제·구독·결제 등 모든 호출이 선택된 서비스의 키로 서명된다.

### 4. 인증 실패(401) 시 키 재입력

서버 어드민에서 서비스 키가 재발급된 경우, 다음 API 호출 시 `401` 응답이 반환된다.

1. 화면에 **"이 서비스의 key가 변경되었습니다. 다시 입력하세요."** 경고 메시지가 표시된다.
2. 자동으로 `/services?reauth=<service_id>`로 리다이렉트된다.
3. 해당 서비스 행이 **빨간 테두리로 강조**되고 키 입력 폼이 펼쳐진 상태로 표시된다.
4. 새 키를 입력해 저장하면 즉시 정상 사용 가능하다.

### 5. 전체 서비스 테스트 절차

```text
1. /services 에서 서비스 목록 확인
2. 미저장 서비스 → api_key + hmac_secret 입력 → [저장 후 선택]
3. 상단 내비에서 활성 서비스명 확인 → /plans → 구독 테스트 수행
4. 상단 내비 [서비스 변경] → 다른 서비스 [선택] → 동일 테스트 반복
5. 키 변경 후 401 발생 시 → reauth 화면에서 새 키 입력
```

> 서비스가 1개이고 .env에 키를 설정한 경우, `/services` 진입 없이도 동작한다(settings 폴백).

| # | 기능 | 방법 |
|---|------|------|
| 1 | 카드 등록 | `/card` → 카드 등록창(또는 수동 authKey) — `register_card()` (`POST /api/v1/cards`) |
| 2 | 카드 조회 | `/card` 진입 시 — `get_card()` (`GET /api/v1/cards/{uid}`) |
| 3 | 카드 변경(재등록) | `/card` 또는 `/my` → 카드 변경 → 카드 등록창 재진입 — `register_card()` (재호출=교체) |
| 4 | 카드 삭제 | `/card` → 카드 삭제 — `delete_card()` (`DELETE`; 사용 중이면 409) |
| 5 | 요금제 목록 조회 | `/plans` 접속 — `get_plans()` |
| 6 | 구독 생성 (일반) | 카드 등록 후 요금제 선택 → "구독하기" — `create_subscription()` (authKey 불필요) |
| 7 | 체험(TRIAL) 시작 | 카드 등록 후 "무료 체험" — `create_subscription(trial=True)` |
| 8 | 구독 조회 | `/my` — `get_subscription()` |
| 9 | 구독 취소 | `/my` → 취소 — `cancel()` |
| 10 | 구독 재개 | `/my` → 재개 — `resume()` |
| 11 | 수동 결제 — SUSPENDED | `/my`에서 상태 SUSPENDED 시 수동 결제 버튼 → `manual_pay()` |
| 12 | 수동 결제 — PAST_DUE | `/my`에서 상태 PAST_DUE(연체) 시 수동 결제 버튼 → `manual_pay()` |
| 13 | 단건(일반) 결제 | `/pay` → 상품명·금액 입력 → 등록 카드로 즉시 결제 — `create_one_off_payment()` (authKey 불필요) |
| 14 | 단건 결제 취소 | `/history` → 미취소 건 취소 버튼 — `cancel_one_off_payment()` |
| 15 | 결제 내역 조회 | `/history` — `get_payments()` (구독 내역) + `OneOffRecord` (단건 내역) 통합 표시 |
