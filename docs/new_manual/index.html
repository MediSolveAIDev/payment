01. 첫날 — 셋업하고 직접 띄워보기

목표: 퇴근 전까지 ① 서버 기동 ② 어드민 로그인 ③ 샘플 쇼핑몰에서 실제 구독 1건 생성 ④ 전체 테스트 통과 — 네 가지를 내 PC에서 확인한다.

1. 사전 준비

  • Docker Desktop (PostgreSQL/Redis 컨테이너용)
  • uv (파이썬 패키지/실행 관리자) — curl -LsSf https://astral.sh/uv/install.sh | sh
  • 토스페이먼츠 테스트 키(test_sk_…, test_ck_…) — 팀 공유 계정의 개발자센터에서 확인

2. 인프라 기동 + 환경 설정

docker compose up -d        # PostgreSQL(127.0.0.1:5433) + Redis(127.0.0.1:6380)
cp .env.example .env.dev    # 환경별 설정 — 아래 두 값은 반드시 채울 것

.env.dev에서 채울 핵심 값:

# AES-256 암호화 키(빌링키·HMAC 시크릿 보관용) — 아래 명령으로 생성
# python3 -c "import os,base64; print(base64.b64encode(os.urandom(32)).decode())"
ENCRYPTION_KEY=<생성한 base64 32바이트>
TOSS_SECRET_KEY=test_sk_...

환경 분리: 공통 .env → 환경별 .env.dev/.env.prod가 덮어쓴다. 실행 환경은 OS 환경변수 APP_ENV(기본 dev)로 정해진다.

3. DB 마이그레이션 + 관리자 계정

uv run alembic upgrade head                 # 전체 테이블 생성
uv run python -m app.cli create-admin --email admin@medisolveai.com --password '<10자 이상>'

4. 서버 기동 + 어드민 접속

uv run uvicorn app.main:app --reload --port 8000
  • 어드민 콘솔: http://127.0.0.1:8000/admin — 방금 만든 계정으로 로그인
  • 헬스체크: http://127.0.0.1:8000/health
  • 데모 데이터가 필요하면: uv run python scripts/seed_demo.py (DEMO-* 서비스 생성 — 운영 DB 금지)

5. 샘플 쇼핑몰로 실제 구독 만들어보기

어드민에서 서비스 등록(허용 IP 127.0.0.1) → 키 화면에서 API 키/HMAC Secret 복사 → 요금제 1개 생성. 그 다음:

cd sample_service
cp .env.example .env && python3 -m venv .venv && .venv/bin/pip install -r requirements.txt
.venv/bin/python manage.py migrate
.venv/bin/python manage.py runserver 8001

http://127.0.0.1:8001 접속 → 서비스 선택(복사한 키 입력) → 이메일 로그인 → 요금제 → 카드 등록(토스 테스트 모드 — 실제 청구 없음) → 어드민의 구독/결제 메뉴에서 방금 만든 구독을 확인한다. 샘플 화면마다 하단 「개발자 노트」가 그 화면의 API·코드를 설명한다 — 첫 주 학습 자료로 가장 좋다.

6. 테스트 돌려보기

uv run pytest          # 전체 (docker compose가 떠 있어야 함 — 테스트 전용 DB/Redis 사용)
uv run pytest -q tests/unit            # 빠른 단위 테스트만

⚠️ 전체 테스트는 동시에 두 개 띄우지 마라. 두 실행이 같은 테스트 DB(payment_test)를 쓰며 시작 시 스키마를 새로 만들기 때문에 서로를 파괴해 수십 건의 가짜 실패가 난다. (09장)

02. 큰그림 — 이 시스템은 무엇이고 어떻게 생겼나

목표: "어떤 요청이 들어오면 어떤 파일을 지나 어디에 기록되는지"를 그림으로 그릴 수 있다.

1. 시스템의 역할

사내 여러 서비스(쇼핑몰, 진료 앱 등)가 공통으로 쓰는 구독·결제 허브다. 각 서비스는 회원/화면만 만들고, 구독 상태와 돈 문제는 전부 이 서버가 책임진다.

사내 서비스 A·B·C ──(HMAC 서명 API)──▶ ┌──────────────────────┐
                                       │  구독·결제 서버        │──▶ 토스페이먼츠
운영자(담당자/관리자) ──(어드민 htmx)──▶ │  FastAPI              │     (빌링키 발급·청구·취소)
                                       │  ├ PostgreSQL (원장)   │
토스 웹훅 ──(IP 검증)─────────────────▶ │  └ Redis (락·세션·캐시)│
                                       └─────────┬────────────┘
                                    APScheduler(5분) — 자동갱신·만료·정산 스윕

네 가지 진입점만 기억하면 된다:

진입점 누가 인증 코드 위치
/api/v1/* 사내 서비스 API키+IP+HMAC 3중 app/api/v1/
/admin/* 운영자 세션 쿠키+CSRF app/admin/
/api/v1/webhooks/toss 토스 발신 IP 검증 app/api/v1/webhooks.py
(HTTP 아님) 스케줄러 시스템 — app/scheduler/runner.py

2. 디렉터리 지도

app/
├── main.py          앱 조립(create_app) — 의존성 주입·라우터 등록·보안헤더
├── core/            설정(config)·DB엔진(db)·암호화(crypto)·에러(errors)·공통 의존성(deps)
├── models/          SQLAlchemy 모델 + enums(상태 정의의 단일 출처)
├── schemas/api.py   외부 API 요청/응답 스키마 (ORM 비노출)
├── api/             외부 API — deps.py(3중 인증), v1/(엔드포인트)
├── admin/           어드민 — routes/(화면), templates/(htmx), filters·pagination(공용)
├── services/        ★ 비즈니스 로직 전부 — 구독·결제·갱신·정산·전이·감사
├── toss/            토스 클라이언트(client) + 테스트용 가짜(fake)
├── scheduler/       5분 배치 러너(전역 락+heartbeat)
└── notifications/   이메일 발송
tests/               unit / integration / e2e / security (09장)
alembic/versions/    DB 마이그레이션 (05장)
sample_service/      외부 서비스 연동 샘플 (Django)

3. 요청의 일생 ① — 외부 서비스가 구독을 만들 때

POST /api/v1/subscriptions
 1) app/api/deps.py        authenticate_service — 킬스위치 → API키 → IP → rate limit
                            → 타임스탬프 → HMAC 서명 → nonce (하나라도 실패 시 즉시 거부)
 2) app/api/v1/subscriptions.py   라우트 — 스키마 검증 후 서비스 함수 호출만
 3) app/services/subscriptions.py create_subscription
      검증 → 토스 빌링키 발급 → Subscription+PENDING Payment 생성 → 1차 commit
      → 토스 첫 결제 → 결과 확정 → 2차 commit + 감사 로그
 4) app/schemas/api.py     SubscriptionResponse.from_model — 민감정보 제외 응답

4. 요청의 일생 ② — 운영자가 어드민에서 목록을 볼 때

GET /admin/subscriptions
 1) app/admin/deps.py      세션 쿠키 → Redis 세션 → 역할/담당 서비스 스코프 결정
 2) app/admin/routes/subscriptions.py   PageParams(검색·정렬·필터 파싱)
 3) app/admin/filters.py   subscription_query — 목록·엑셀·탭이 공유하는 단일 쿼리 빌더
 4) app/admin/pagination.py paginate — count·페이징 자동 처리
 5) templates/…            전체 페이지 또는 htmx partial(_table.html)만 렌더

5. 레이어 규칙 — 어기면 안 되는 4가지

  1. 라우트는 얇게: 파싱·렌더만. 비즈니스 규칙은 전부 app/services/에.
  2. commit은 서비스 레이어가 한다. 라우트에서 commit하지 않는다.
  3. 오류는 app/core/errors.py의 DomainError 계열을 던진다 — HTTP 변환은 핸들러가 자동으로.
  4. services는 api/admin을 import하지 않는다 (방향: 라우트 → 서비스 → 모델).

03. 도메인 — 구독과 결제가 움직이는 규칙

목표: 상태머신과 결제 3원칙을 설명할 수 있다. 이 장이 이 시스템의 심장이다.

1. 용어 사전

용어 의미 비고
Service 이 서버를 쓰는 사내 서비스(테넌트) API키·HMAC 시크릿·허용 IP 보유
Plan 요금제 — 가격·주기(년/월/주/일)·할인·체험 결제 주기는 생성 후 수정 불가
Subscription (서비스, 사용자)의 구독 1건 서비스+사용자당 열린 구독 1개 규칙
Payment 개별 결제 시도 기록 — 성공/실패 모두 보존 금액은 전부 정수 KRW
external_user_id 외부 서비스 쪽 사용자 ID 문자열 이 서버엔 회원 개념이 없다
customer_key / billing key 토스 고객키 / 자동결제 키 빌링키는 AES 암호화 보관, 평문 저장 금지

2. 구독 상태머신 — 단일 출처는 app/services/transitions.py

TRIAL ──▶ ACTIVE ──▶ PAST_DUE ──▶ SUSPENDED ──▶ EXPIRED(종단)
  │          │ ▲        │ ▲           │
  │          │ └────────┘ └─(수동결제)─┴──▶ ACTIVE
  └────┬─────┴─────────────┘
       ▼
   CANCELED ──▶ EXPIRED          (만료 전 재개: CANCELED → ACTIVE | PAST_DUE)
   EXTENDED(운영자 만료일 연장) — 활성과 동등, 새 만료일에 자동갱신   ※ 최근 추가
상태 뜻 접근 자동결제
TRIAL 체험 중(결제 없음) O 체험 만료일에 첫 결제
ACTIVE 정상 O 주기마다 갱신
PAST_DUE 갱신 실패, 재시도 중(미수) O 12시간 간격 최대 4회(전역설정)
SUSPENDED 재시도 소진 → 정지 X 중지 — 수동결제로만 복구
CANCELED 해지 예약(만료일까지 이용) O 중지 — 만료 전 재개 가능
EXTENDED 운영자가 만료일 수동 연장 O 새 만료일에 갱신
EXPIRED 완전 종료(종단 — 어떤 전이도 불가) X —

규칙: 상태를 바꿀 땐 반드시 transition(sub, 새상태)를 호출한다. 허용 전이 테이블(ALLOWED_TRANSITIONS)에 없는 전이는 예외로 즉시 터지고, 보편 불변식(EXPIRED/CANCELED→자동결제 중지, SUSPENDED→정지시각 기록, ACTIVE 복귀→실패 흔적 초기화)이 자동 적용된다. sub.status = ... 직접 대입 금지.

3. 결제 3원칙 — 절대 깨지 마라

이 세 가지가 이중결제를 막는 전부다. 결제 코드를 고칠 때마다 자문하라: "이 변경이 3원칙을 깨는가?"

  1. PENDING 선커밋 — 토스를 호출하기 전에 PENDING Payment를 commit한다. 서버가 죽어도 "돈이 나갔을 수 있다"는 기록이 남아 정산 스윕이 확정할 수 있다.
  2. 타임아웃은 실패가 아니다 — 토스 응답을 못 받으면 결과 불명이다. FAILED로 확정하는 순간, 실제론 승인된 결제 위에 재결제가 얹혀 이중결제가 된다. PENDING 유지 → 10분 후 정산 스윕(reconciliation.py)이 토스 재조회로 확정.
  3. 멱등 order_id — 갱신 결제는 (구독, 기간, 시도번호)로 만든 결정적 order_id를 쓴다. 같은 주문을 재시도하면 토스가 멱등키로 같은 결과를 재생한다.

4. 돈 계산 규칙 — 단일 출처는 app/services/billing_math.py

  • 금액은 전부 정수(KRW). float 금지. 나눗셈은 //(내림 — 항상 고객에게 유리).
  • 첫 결제액 = 정가 + 첫구독 할인(FREE/정액/정률) — 첫구독에만.
  • 정기 결제액 = 정가 + 상시 할인 — 2회차부터, 그리고 체험 전환 첫 결제도 이 금액.
  • 취소 수수료 = 금액 × 수수료% // 100, 환불액 = 금액 − 수수료. (fee+refund==amount 항상 성립)
  • 기간 계산(compute_period_end)은 relativedelta — 1/31 + 1개월 = 2/28(월말 클램프).

5. 그 밖의 핵심 규칙

  • 서비스+사용자당 열린 구독 1개 — DB 부분 유니크 인덱스가 최종 심판(EXPIRED만 제외 → 재구독 허용).
  • 자동결제 안함(auto_renew=False) 요금제 — 첫 결제 후 다음 결제를 예약하지 않고 기간 종료 시 만료.
  • 첫구독 혜택은 1회 — "결제를 소진한(또는 무료로 활성화된) 과거 구독"이 있으면 재구독은 혜택 없음.
  • 접근 판단은 access_allowed 하나로 — 외부 서비스는 상태별 분기를 직접 만들지 말 것.
  • 요금제·서비스는 구독이 있으면 삭제 불가 (DB FK RESTRICT + 서비스 레이어 검증).

06. 인증·보안 — 세 개의 문과 그 열쇠

목표: 401/403이 났을 때 어느 단계에서 막혔는지 바로 짚을 수 있다.

1. 외부 API — 3중 인증 (app/api/deps.py authenticate_service)

요청 헤더 4개(x-service-key·x-timestamp·x-nonce·x-signature)로 6단계 검증:

0 킬스위치(전역설정) → 1 API키 해시 대조 → 2 IP 화이트리스트 → 3 rate limit
→ 4 타임스탬프 ±300초 → 5 HMAC 서명(본문 포함) → 6 nonce 1회용

서명 = HMAC_SHA256(secret, "METHOD\n경로\n타임스탬프\nnonce\nsha256(본문)"). 클라이언트 구현은 sample_service/shop/payment_client.py를 복사하는 게 정답 — 서버 구현(app/core/security.py:sign_request)의 검증된 미러다.

401 디버깅 순서: ① 서버 시계 오차(±300초) ② 키 재발급 여부(어드민에서 rotate하면 즉시 옛 키 무효) ③ nonce 재사용 ④ 본문 그대로 서명했는지(JSON 직렬화 차이).

2. 어드민 — 세션 + CSRF

  • 로그인 실패 5회 → 15분 잠금. IP당 분당 10회 rate limit(무차별 시도 차단).
  • 세션: Redis, 유휴 30분 연장형 + 절대 수명 12시간(연장돼도 12시간 후 재로그인).
  • 모든 POST는 CSRF 토큰 검증. 역할: SYSTEM_ADMIN(전체) / SERVICE_MANAGER(담당 서비스만 — 남의 리소스는 403이 아니라 404로 숨긴다).
  • 어드민 접속 IP 제한은 전역설정에서 — 현재 IP를 빼고 저장하면 잠기므로 검증이 막아준다.

3. 운영 장치

장치 무엇 어디서
킬스위치 외부 API 전체를 503으로 차단(어드민은 영향 없음) 어드민 → 전체설정 → 서버 비활성화(본인 비밀번호 재확인). 5초 TTL Redis 캐시 — 전환 시 즉시 무효화
키 회전 서비스 API키/HMAC 즉시 재발급(옛 키 무효) 서비스 상세 → 재발급. 외부 서비스에 새 키 전달 필수
감사 로그 누가 언제 무엇을 — 모든 변경 기록 어드민 → 감사 로그(검색·엑셀)
웹훅 IP 검증 토스 공식 발신 IP만 허용 WEBHOOK_IP_CHECK_ENABLED (dev에선 끄고 테스트)

4. 운영 배포 체크리스트 (prod 반영 전 반드시)

  1. APP_ENV=prod 설정 — dev 로그인 프리필 차단 + HSTS 활성화 조건
  2. TRUST_PROXY=true면 TRUST_PROXY_HOPS를 실제 프록시 단 수로 — XFF는 "오른쪽에서 n번째"를 신뢰한다(왼쪽은 위조 가능). 잘못 설정하면 정상 요청이 거부됨(안전한 방향)
  3. 인터넷 직노출이면 PUBLIC_SERVICE_LIST_ENABLED=false (무인증 서비스 목록 차단)
  4. uv run alembic upgrade head
  5. docker-compose.yml은 개발 전용 — 운영은 별도 자격증명의 관리형 DB/Redis

07. 배치와 장애 대응 — 새벽에 전화 왔을 때 보는 장

목표: "결제가 이상해요"라는 말을 들었을 때 어디부터 볼지 안다.

1. 5분 배치가 하는 일 (app/scheduler/runner.py → services/renewals.py)

run_renewals (전역 Redis 락 — 토큰+heartbeat로 중복 실행 방지)
 └ process_due — 4종 대상을 due 오름차순으로 최대 1000건씩, 병렬 10으로 처리
    ├ 갱신 결제   TRIAL/ACTIVE/PAST_DUE/EXTENDED + next_billing_at 도래 → 결제·기간 전진
    ├ 취소 만료   CANCELED + 기간 종료 → EXPIRED + 빌링키 삭제
    ├ 정지 만료   SUSPENDED + 유예(기본 30일) 초과 → EXPIRED + 빌링키 삭제
    ├ 비자동갱신  ACTIVE + 자동결제 없음 + 기간 종료 → EXPIRED
    └ 정산 스윕   10분 넘게 PENDING인 결제를 토스 재조회로 DONE/FAILED 확정
  • 갱신 1건 = Redis 구독 락 → FOR UPDATE 검증+PENDING 선커밋 → (락/커넥션 반납 후) 토스 호출 → FOR UPDATE 재취득·재검증 → 확정. 이 3단계 구조를 바꾸지 마라 — 토스 지연 시 서버 전체가 멈추는 것을 막는 장치다.
  • 재시도 정책(횟수·간격·유예)은 코드가 아니라 어드민 전체설정에서 바꾼다(다음 배치부터 적용).

2. 증상별 대응 — 순서대로 본다

"결제했는데 반영이 안 됐대요" (PENDING에 멈춤)

  1. 어드민 → 결제에서 해당 건 상태 확인. PENDING이면 정상 동작 중 — 토스 타임아웃이고 10분 내 정산 스윕이 확정한다. 절대 수동으로 FAILED 처리하지 마라(이중결제 위험).
  2. 15분 지나도 PENDING이면: 서버 로그에서 payment.reconciliation 항목 확인, 토스 개발자센터에서 해당 주문(toss_order_id) 실제 상태 대조.

"구독이 정지됐어요" (SUSPENDED)

  • 카드 한도/정지 등으로 재시도 4회 실패한 것. 사용자에게 카드 변경 후 수동 결제 안내(샘플 /my 화면 또는 외부 서비스의 수동결제 API). 어드민 구독 상세의 "재결제" 버튼으로 운영자가 대신 시도할 수도 있다.
  • 유예(기본 30일) 안에 결제 없으면 자동 EXPIRED — 그 후엔 재구독만 가능.

"토스가 장애래요"

  • 아무것도 끄지 마라. 타임아웃→PENDING→스윕 확정 설계가 알아서 수습한다.
  • 장애가 길어지면(수 시간) 갱신 대상이 적체된다 — 복구 후 배치가 5분마다 1000건씩 자동 소화한다. 로그의 배치 상한 도달 경고는 적체 소화 중이라는 뜻.

"외부 서비스가 전부 401이래요"

  • 어드민에서 그 서비스 키를 회전했는지 확인(감사 로그 service.rotate_keys). 새 키를 전달하면 끝. 아니면 06장의 401 디버깅 순서.

긴급 전체 차단이 필요할 때

  • 어드민 → 전체설정 → 서버 비활성화(사유 입력). 외부 API만 503, 어드민은 살아있다. 해제도 같은 화면. 모든 전환이 감사 로그에 남는다.

3. 어디를 보나 (관측 지점)

보고 싶은 것 위치

09. 테스트 — 안전망 사용법

목표: 변경 후 무엇을 돌려야 하는지, 새 테스트를 어디에 어떻게 쓰는지 안다. 이 프로젝트의 1차 안전망은 실제 Postgres/Redis + 가짜 토스로 도는 통합 테스트다.

1. 구조

디렉터리 무엇 예시
tests/unit DB 없는 순수 로직 billing_math, transitions(상태머신 규칙), XFF 파싱
tests/integration 서비스 함수 + 실제 DB/Redis test_renewals.py(갱신 상태머신 — 가장 중요), 구독 생성/관리, 정산, 웹훅
tests/e2e HTTP로 화면/API 전체 어드민 화면, htmx, 엑셀, 외부 API
tests/security 인증 전용 HMAC 6단계, 어드민 보안

2. 인프라가 어떻게 도는가 (tests/conftest.py)

  • docker compose up -d의 같은 컨테이너를 쓰되 별도 DB(payment_test)·Redis DB 15 사용
  • 세션 시작 시 drop_all → create_all(마이그레이션 미사용 — 모델이 곧 스키마), 테스트마다 TRUNCATE + Redis flush — 테스트 간 독립
  • ⚠️ 같은 머신에서 pytest 두 개를 동시에 돌리지 마라 — 서로의 스키마를 drop해서 수십 건의 가짜 실패(IntegrityError 등)가 난다. 진짜 실패인지 헷갈리면 단독으로 재실행이 1순위.

3. FakeTossClient — 결제 테스트의 핵심 (app/toss/fake.py)

실제 토스 대신 주입되는 가짜. 시나리오 주입이 가능하다:

fake = FakeTossClient()
fake.fail_charge_with = TossError("REJECT_CARD", "카드 거절")   # 다음 결제를 거절
fake.fail_charge_with = TossTimeoutError()                      # 타임아웃(결과불명) 시뮬레이션
fake.payments_by_order[toss_order_id] = FakeTossClient._result_for(toss_order_id, 5000)
# ↑ "토스에는 사실 승인돼 있었다" 상태를 만들어 정산 스윕 수렴을 테스트
fake.charges          # 토스에 도달한 청구 기록(금액·멱등키 검증용)

멱등키 재생(같은 키 재요청 시 같은 결과 반환)까지 재현하므로, 이중결제 방어 테스트는 fake.charges 길이로 검증하는 패턴을 쓴다.

4. 자주 쓰는 명령

uv run pytest -q                          # 전체 (≈50초)
uv run pytest tests/integration/test_renewals.py -q     # 갱신 상태머신만
uv run pytest -k "manual" -q              # 이름 매칭
uv run pytest --lf -x                     # 직전 실패만, 첫 실패에서 중단

  • 픽스처: db(세션) cipher redis_client client(HTTP) + tests/factories.py (create_service/plan/subscription/user)부터 시작

10. 트러블슈팅 FAQ — 증상으로 찾기

서버가 안 뜬다

  • ENCRYPTION_KEY 미설정/형식 오류 — 기동 시 가장 먼저 검증한다. .env.dev에 base64 32바이트인지 확인(01장 생성 명령).
  • 포트 5433/6380 연결 실패 — docker compose up -d 했는지, 다른 프로세스가 점유했는지.

alembic upgrade head가 실패한다

  • Multiple heads: 두 갈래 리비전 — uv run alembic heads로 확인 후 down_revision 체인 수정.
  • 이미 수동으로 스키마를 바꾼 DB라면: 마이그레이션과 실제 스키마 차이를 확인하고 맞춰라(테스트 DB는 그냥 지워도 됨 — 다음 pytest가 재생성).

외부 서비스가 401을 받는다

06장의 순서대로: 시계 오차(±300초) → 키 회전 여부(감사 로그 service.rotate_keys) → nonce 재사용 → 서명 본문 불일치. 빠른 검증법: sample_service에 같은 키를 넣고 요금제 조회가 되는지 — 되면 키는 정상이고 상대 구현 문제.

결제는 됐다는데 DB에 PENDING이다

정상이다. 토스 타임아웃 → 결과불명 → 10분 후 정산 스윕이 토스 재조회로 확정한다. 절대 수동으로 FAILED 처리하지 마라(07장). 15분 넘게 안 풀리면 로그 + 토스 콘솔 대조.

어드민에서 어떤 항목이 404가 난다 (분명히 존재하는데)

의도된 동작일 수 있다 — SERVICE_MANAGER는 담당 외 서비스 리소스를 404로 숨긴다 (존재 비노출). SYSTEM_ADMIN 계정으로 보이면 권한 스코프 문제다.

테스트가 수십 개씩 무더기로 실패한다

다른 pytest가 동시에 돌고 있는지부터 의심(09장) — 단독 재실행이 1순위 진단. 단독에서도 실패하면 그때가 진짜 버그다.

manual.html을 고쳤는데 다음 빌드에서 사라졌다

manual.html/index.html은 생성물이다. 원본 마크다운(또는 build_html.py)을 고치고 재빌드하라. (dev_manual·new_manual 동일)

샘플 쇼핑몰이 "구독서버에 연결할 수 없습니다"라고 한다

구독서버(:8000)가 떠 있는지 → 샘플의 서비스 키가 맞는지(어드민 키 화면과 대조) → 서비스 허용 IP에 127.0.0.1이 있는지.

갱신이 안 도는 것 같다

  • .env의 SCHEDULER_ENABLED/SCHEDULER_INTERVAL_MINUTES 확인.
  • 로그에 renewal batch skipped — 다른 인스턴스가 실행 중이 계속 찍히면: 죽은 프로세스의 전역 락이 남은 것 — Redis에서 lock:scheduler:renewals TTL(최대 240초) 후 자연 해소.
  • 수동 1회 실행으로 확인: 테스트의 run_renewals(app) 패턴 참고.

이메일이 안 나간다

GMAIL_ID/GMAIL_PW(앱 비밀번호) 미설정이면 콘솔 출력 모드가 기본 — 로그에서 메일 내용을 확인할 수 있다. 실제 발송은 두 값을 설정해야 한다.

11. 알려진 이슈와 정책 결정 대기 — 코드 고치기 전에 읽어라

이 저장소는 2026-06에 보안·성능·구조 감사(4 Phase 적용 완료)와 도메인 로직 검증을 거쳤다. 적용이 끝난 것과 알고도 남겨둔 것을 구분해서 인수한다. 원문: 감사 리포트 · 로직 검증 리포트 · Phase별 워크로그(같은 폴더)

1. 수정 권장 — 아직 미적용 (로직 검증 High)

손대기 전에 반드시 원문 리포트의 재현 시나리오를 읽을 것.

# 문제 한 줄 요약
H-1 수동 결제 이중 청구 수동 결제가 구독 락 없이 매번 새 멱등키로 토스 호출 — 배치와 겹치거나 더블클릭이면 2회 청구
H-2 승인된 돈 영구 미기록 정산 스윕이 수동 결제 PENDING을 "배치가 처리할 것"으로 오인해 영원히 건너뜀
H-3 재시도 중 가격 변경 시 금액 불일치 PENDING 재사용 시 청구액을 재계산하지만 DB 기록액은 안 바꿈
H-4 100% 상시할인 = 구독 자동 파괴 0원 갱신 청구 가드 없음 — 토스 거절 반복 → 정지 → 만료

2. 정책 결정이 먼저 필요한 것 (코드보다 의사결정)

주제 현재 동작 결정할 것
환불 월 귀속 원결제 월에 소급 반영(canceled_at 미사용) 취소 시점 월로 집계할지
월말 앵커 1/31 가입 → 영원히 28일로 드리프트 가입일 유지 정책으로 바꿀지, 현행 문서화할지
체험 후 첫구독 혜택 체험만 쓴 사용자는 할인 영구 상실 의도(악용 방지)인지 확인·문서화
