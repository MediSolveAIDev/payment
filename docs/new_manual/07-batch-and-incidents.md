# 07. 배치와 장애 대응 — 새벽에 전화 왔을 때 보는 장

> 목표: "결제가 이상해요"라는 말을 들었을 때 어디부터 볼지 안다.

## 1. 5분 배치가 하는 일 (app/scheduler/runner.py → services/renewals.py)

```
run_renewals (전역 Redis 락 — 토큰+heartbeat로 중복 실행 방지)
 └ process_due — 4종 대상을 due 오름차순으로 최대 1000건씩, 병렬 10으로 처리
    ├ 갱신 결제   TRIAL/ACTIVE/PAST_DUE/EXTENDED + next_billing_at 도래 → 결제·기간 전진
    ├ 취소 만료   CANCELED + 기간 종료 → EXPIRED + 빌링키 삭제
    ├ 정지 만료   SUSPENDED + 유예(기본 30일) 초과 → EXPIRED + 빌링키 삭제
    ├ 비자동갱신  ACTIVE + 자동결제 없음 + 기간 종료 → EXPIRED
    └ 정산 스윕   10분 넘게 PENDING인 결제를 토스 재조회로 DONE/FAILED 확정
```

- 갱신 1건 = Redis 구독 락 → FOR UPDATE 검증+PENDING 선커밋 → **(락/커넥션 반납 후)**
  토스 호출 → FOR UPDATE 재취득·재검증 → 확정. **이 3단계 구조를 바꾸지 마라** —
  토스 지연 시 서버 전체가 멈추는 것을 막는 장치다.
- 재시도 정책(횟수·간격·유예)은 코드가 아니라 **어드민 전체설정**에서 바꾼다(다음 배치부터 적용).

## 2. 증상별 대응 — 순서대로 본다

### "결제했는데 반영이 안 됐대요" (PENDING에 멈춤)
1. 어드민 → 결제에서 해당 건 상태 확인. PENDING이면 **정상 동작 중** —
   토스 타임아웃이고 10분 내 정산 스윕이 확정한다. **절대 수동으로 FAILED 처리하지 마라**(이중결제 위험).
2. 15분 지나도 PENDING이면: 서버 로그에서 `payment.reconciliation` 항목 확인,
   토스 개발자센터에서 해당 주문(toss_order_id) 실제 상태 대조.

### "구독이 정지됐어요" (SUSPENDED)
- 카드 한도/정지 등으로 재시도 4회 실패한 것. 사용자에게 **카드 변경 후 수동 결제**
  안내(샘플 `/my` 화면 또는 외부 서비스의 수동결제 API). 어드민 구독 상세의
  "재결제" 버튼으로 운영자가 대신 시도할 수도 있다.
- 유예(기본 30일) 안에 결제 없으면 자동 EXPIRED — 그 후엔 재구독만 가능.

### "토스가 장애래요"
- 아무것도 끄지 마라. 타임아웃→PENDING→스윕 확정 설계가 알아서 수습한다.
- 장애가 길어지면(수 시간) 갱신 대상이 적체된다 — 복구 후 배치가 5분마다
  1000건씩 자동 소화한다. 로그의 `배치 상한 도달` 경고는 적체 소화 중이라는 뜻.

### "외부 서비스가 전부 401이래요"
- 어드민에서 그 서비스 키를 회전했는지 확인(감사 로그 `service.rotate_keys`).
  새 키를 전달하면 끝. 아니면 06장의 401 디버깅 순서.

### 긴급 전체 차단이 필요할 때
- 어드민 → 전체설정 → **서버 비활성화**(사유 입력). 외부 API만 503, 어드민은 살아있다.
  해제도 같은 화면. 모든 전환이 감사 로그에 남는다.

## 3. 어디를 보나 (관측 지점)

| 보고 싶은 것 | 위치 |
|---|---|
| 배치 실행 결과 | 서버 로그 `renewal batch done: {renewed: …, failed: …}` |
| 개별 실패 사유 | 결제 상세의 failure_code/메시지 + 감사 로그(action으로 검색) |
| 돈 흐름 요약 | 대시보드(이번 달 매출·환불·미수) / 정산 화면(월별) |
| 시스템이 한 일 전부 | 감사 로그 — actor_type=SYSTEM이 배치의 행동 |
