# 인수인계 가이드 — 처음 맡는 개발자를 위한 안내

> 이 매뉴얼은 **이 프로젝트를 처음 맡는 개발자가 혼자서 운영·유지보수·기능추가까지
> 할 수 있게 되는 것**을 목표로 한다. 기능별 상세 레퍼런스인
> [개발자 매뉴얼(dev_manual)](../dev_manual/manual.html)과 달리, **읽는 순서가 있는 학습 경로**로 구성했다.

## 어떻게 읽어야 하나

| 시점 | 읽을 장 | 목표 |
|---|---|---|
| **첫날** | 01 | 내 PC에서 서버·어드민·샘플을 실제로 띄우고 테스트를 돌린다 |
| **첫 주** | 02 → 03 → 04 | 시스템 큰그림, 도메인 규칙(상태머신·결제 3원칙), 코드 지도를 머리에 넣는다 |
| **필요할 때** | 05 · 06 · 07 · 09 · 10 | DB/인증/배치/테스트/트러블슈팅 — 사전처럼 찾아본다 |
| **기능을 추가할 때** | 08 | 실제 변경 작업의 단계별 레시피를 그대로 따라 한다 |
| **코드를 고치기 전에 반드시** | 11 | 알려진 이슈·정책 결정 대기 항목 — 모르고 건드리면 다치는 곳 |

## 3가지 대원칙 (이것만은 외우자)

1. **코드가 정답이다.** 문서와 코드가 다르면 코드가 맞다. 문서를 고치고
   `uv run --with markdown python docs/new_manual/build_html.py`로 재생성하라.
2. **결제 코드는 "결제 3원칙"을 절대 깨지 마라.** (03장) — PENDING 선커밋,
   타임아웃은 실패가 아님, 멱등 order_id. 이 원칙이 이중결제를 막는 전부다.
3. **변경에는 흔적을 남겨라.** 주석/docstring 갱신 → 테스트 → 매뉴얼 반영 →
   `docs/audit/`에 날짜별 워크로그. 이 프로젝트의 모든 변경이 이 규칙으로 기록되어 있다.

## 다른 문서와의 관계

- **[dev_manual (manual.html)](../dev_manual/manual.html)** — 기능별 상세 레퍼런스(요청→파일→DB→응답 추적). 이 가이드가 "지도"라면 dev_manual은 "백과사전".
- **[docs/audit/](../audit/)** — 보안·성능·구조 감사와 로직 검증 리포트, 변경 워크로그. 11장에서 요약한다.
- **[sample_service/](../../sample_service/README.md)** — 외부 서비스 연동 샘플(서비스팀 제공용). 화면마다 「개발자 노트」가 붙어 있어 API 학습에 가장 빠르다.
- **docs/toss/** — 토스페이먼츠 연동 참고 문서.
