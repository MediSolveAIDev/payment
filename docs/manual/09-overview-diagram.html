구독/결제 서버가 어떻게 동작하는지를 그림으로 정리했습니다. 이 페이지만 보고도 전체 그림이 잡히도록 구성했습니다.

🖱️

그림 속 박스·카드를 누르면 관련 매뉴얼 장으로 이동합니다(마우스를 올리면 파란 테두리로 표시). 각 그림 아래에는 상세 설명과 🔗 관련 페이지 링크도 함께 있습니다.

A.시스템 구성도 — 누가 누구와 통신하나

핵심은 셋입니다. 우리 서비스가 구독/결제 서버에 API로 요청하고, 결제 서버가 토스페이먼츠와 결제를 처리합니다. 카드 번호는 토스에만 입력됩니다.

그림의 우리 서비스·관리자 콘솔 박스(↗)를 누르면 해당 장으로 이동합니다.

각 구성요소가 하는 일

  • 사용자 — 우리 서비스의 화면(앱·웹)을 이용하는 실제 고객입니다. 카드 등록 시에만 토스 결제창과 직접 통신하며, 카드번호는 우리 서버나 결제 서버를 거치지 않습니다.
  • 우리 서비스(백엔드) — 결제 서버에 HMAC 인증 API로 구독 생성·조회·결제를 요청합니다. 연동 방법은 7. API 연동에서 실제 코드로 다룹니다.
  • 구독/결제 서버 — 요금제·구독·결제를 관리하고, 내부 스케줄러가 만료일마다 자동 결제를 청구합니다. 데이터는 PostgreSQL·Redis에 보관됩니다.
  • 토스페이먼츠(PG) — 실제 카드 승인·빌링키 발급을 담당합니다. 결제 서버와 서버 간 통신하며, 결제 이벤트를 웹훅으로 알려줍니다.
  • 관리자 콘솔 — 서비스 담당자가 로그인해 요금제·구독·결제·정산을 관리하는 웹 화면입니다. 화면별 사용법은 5. 화면별 가이드에 있습니다.

B.연동 준비 — 첫 구독을 받기까지

새 서비스를 결제 서버에 연결해 첫 구독을 받기까지의 순서입니다.

STEP 1
담당자 계정 생성
콘솔 로그인 계정 발급(비밀번호 설정 메일)
→ 5.7 계정 관리
STEP 2
서비스 등록
서비스명·허용 IP·담당자·취소 정책
→ 5.2 서비스 등록
STEP 3
키 발급·전달
API 키·HMAC 시크릿(1회 표시)
→ 5.2.2 키 발급
STEP 4
요금제 생성
가격·주기·할인·체험 설정
→ 1. 요금제 관리
STEP 5
API 연동
자사 서버에서 구독·결제 호출
→ 7. API 연동

각 단계 카드를 누르면 해당 설명으로 이동합니다.

왜 이 순서인가

담당자 계정(1)을 서비스 등록(2)보다 먼저 만들어야 합니다. 서비스 등록 화면에서 담당자를 선택하는 방식이라, 계정이 없으면 담당자를 지정할 수 없기 때문입니다. 키(3)는 발급 순간 한 번만 전체가 표시되므로 즉시 안전하게 보관·전달해야 합니다. 요금제(4)가 최소 1개 활성 상태여야 구독을 받을 수 있고, 그다음 API 연동(5)으로 실제 구독·결제를 호출합니다.

C.구독 시작 흐름 — 카드 등록부터 첫 결제까지

사용자가 카드를 등록하면 토스가 authKey를 주고, 우리 백엔드가 그것으로 구독을 만듭니다(빌링키 발급 + 첫 결제).

단계별로 보면

  • 요금제 조회 — 우리 서비스가 GET /plans로 구독 가능한 요금제와 실제 청구 금액을 받아 사용자에게 보여줍니다.
  • 카드 등록 — 사용자가 토스 SDK requestBillingAuth()로 열린 카드 등록창에 카드를 입력합니다. (카드번호는 토스만 취급)
  • authKey 수신 — 인증이 끝나면 토스가 successUrl로 돌아오며 authKey·customerKey를 전달합니다.
  • 구독 생성 — 우리 백엔드가 그 authKey로 POST /subscriptions를 호출하면, 결제 서버가 빌링키 발급 + 첫 결제를 처리합니다.
  • 접근 판단 — 응답의 access_allowed(true/false) 하나로 사용자 이용 허용 여부를 결정합니다. 상태를 직접 해석할 필요가 없습니다.

이후 만료일마다 결제 서버가 자동으로 정기 결제를 청구합니다(아래 E 참고). 핵심 규칙(금액 변조 방지·멱등 order_id 등)과 실제 코드는 API 연동 장에 있습니다.

D.구독 상태 라이프사이클

구독은 아래 상태들을 오갑니다. 초록은 이용 가능, 빨강은 차단입니다.

이용 가능(access_allowed = true) 이용 차단(false)

각 상태 박스를 누르면 2. 구독 관리로 이동합니다.

상태별 의미

  • TRIAL 체험 중 — 결제 없이 이용, 체험이 끝나면 첫 결제가 발생합니다. 이용 가능
  • ACTIVE 정상 이용 — 결제가 완료된 상태. 만료일마다 자동 갱신됩니다. 이용 가능
  • PAST_DUE 미수 — 결제가 실패했지만 바로 막지 않고 자동 재시도하는 유예 기간입니다. 이용 가능
  • EXTENDED 연장처리 — 관리자가 만료일을 수동 연장한 상태(보상·이벤트 등). 이용 가능
  • CANCELED 해지 예약 — 해지했지만 만료일까지는 유지되고, 만료 시 종료됩니다. 이용 가능
  • SUSPENDED 정지 — 반복 결제 실패로 차단된 상태. 즉시 재결제(수동 결제)로 복구할 수 있습니다. 차단
  • EXPIRED 완전 종료 — 최종 상태로 더 이상 갱신되지 않습니다. 차단

핵심은 결제 한 번 실패 = 즉시 차단이 아니라는 점입니다. 여러 번 재시도하고 유예 기간을 준 뒤에야 정지됩니다(아래 E 참고).

E.자동 갱신과 결제 재시도

만료일이 되면 서버 스케줄러가 자동으로 정기 결제를 청구합니다. 실패하면 정해진 횟수만큼 재시도합니다.

SUSPENDED 박스를 누르면 2. 구독 관리로 이동합니다.

자동 갱신·재시도가 동작하는 방식

  • 만료일 도달 — 서버 스케줄러가 주기적으로 감지해 자동 결제를 시도합니다(담당자·사용자 개입 불필요).
  • 성공 — 다음 주기로 갱신되고 상태는 ACTIVE로 유지됩니다.
  • 실패 — PAST_DUE(미수)로 두고 정해진 간격으로 재시도합니다(이용은 계속 허용).
  • 재시도 소진 — 끝내 실패하면 SUSPENDED(정지)로 차단됩니다. 이후 유예 기간이 지나면 EXPIRED로 종료됩니다.

재시도 횟수·간격과 정지 후 만료 유예일은 콘솔의 전체 설정에서 조정합니다. 고객이 카드를 정리했다면 다음 자동 시도를 기다리지 않고 구독 상세의 즉시 재결제로 바로 복구할 수 있습니다.
