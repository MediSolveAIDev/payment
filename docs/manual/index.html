구독/결제 서버
서비스 담당자 매뉴얼

사내 서비스에 구독·결제 기능을 붙이는 담당자를 위한 안내서입니다. 관리자 콘솔 사용법부터 자사 서버의 API 연동까지, 처음 시작하는 분도 따라 할 수 있도록 정리했습니다.

🚀

처음이신가요? 0. 시작하기에서 전체 그림과 로그인 방법을 먼저 확인하세요. 이미 콘솔에 익숙하다면 곧장 7. API 연동으로 이동해도 됩니다.

서버를 처음 구축하는 시스템 관리자라면 6. 설치·설정·배포부터 시작하세요(인프라·첫 관리자 계정·토스 연결).

A.이 시스템은 무엇인가요?

사내 여러 서비스가 공통으로 사용하는 구독·결제 인프라입니다. 각 서비스가 직접 결제 모듈(토스페이먼츠)을 연동하지 않아도, 이 서버에 요금제를 등록하고 API를 호출하기만 하면 정기 구독과 단건 결제를 처리할 수 있습니다. 결제 승인·자동 갱신·재시도·정산 집계는 모두 서버가 대신 처리합니다.

우리 서비스에 '넷플릭스 같은 자동 결제'를 직접 개발하지 않고 빌려 쓰는 것이라고 생각하면 됩니다. 손님이 카드를 한 번 등록하면, 그다음부터는 이 서버가 알아서 매달(또는 매년) 돈을 받아 주고, 실패하면 다시 시도하고, 매출도 정리해 줍니다. 우리는 '무엇을 얼마에 팔지'(요금제)만 정해 주면 됩니다.

서비스 담당자는 보통 두 가지 일을 합니다.

역할 ①

관리자 콘솔 운영

웹 화면에 로그인해 요금제를 만들고, 구독·결제 현황을 확인하고, 환불·만료일 조정 등 운영 작업을 합니다.

역할 ②

API 연동

자사 서비스(앱·서버)에서 이 서버의 API를 호출해 실제 구독 신청·결제·해지를 처리하도록 개발팀과 연동합니다.

B.전체 흐름 한눈에 보기

새 서비스를 연결해 첫 구독을 받기까지의 순서입니다. 각 단계의 자세한 방법은 해당 장에서 다룹니다.

STEP 1
계정 발급·로그인
담당자 계정으로 콘솔 로그인
STEP 2
서비스·키 확인
API 키·HMAC 시크릿 수령
STEP 3
요금제 생성

STEP 4
API 연동

STEP 5
운영·정산
현황 모니터링·환불·정산
ℹ️

서비스 등록·키 발급·담당자 계정 생성은 보통 시스템 관리자가 먼저 처리해 두고, 키를 담당자에게 전달합니다. 담당자는 전달받은 키와 계정으로 3단계부터 시작하는 경우가 많습니다.

D.꼭 알아둘 핵심 규칙

  • 서비스 + 사용자당 구독은 1개만 가능합니다. 같은 사용자가 이미 구독 중이면 새 구독 생성이 거부됩니다.
  • 결제 금액은 서버가 계산합니다(요금제 기준). 외부에서 금액을 임의로 보낼 수 없어 조작이 차단됩니다. (단건 결제만 예외적으로 금액 지정)
  • 구독 취소는 즉시 종료가 아니라 만료일 예약 종료입니다. 만료일까지는 이용이 유지됩니다.
  • 구독이 남아 있는 요금제·서비스는 삭제할 수 없습니다.
  • 결제 주기는 요금제 생성 시에만 설정되며 이후 변경할 수 없습니다.
  • API 키·HMAC 시크릿은 발급 시 한 번만 표시됩니다. 즉시 안전하게 보관하세요.

E.용어집

용어의미
서비스(Service)이 인프라를 사용하는 사내 앱/상점 단위. API 키와 허용 IP, 취소 정책을 가집니다.
요금제(Plan)판매할 구독 상품. 가격·주기·할인·체험 정책을 담습니다.
구독(Subscription)한 사용자가 한 요금제를 이용 중인 계약. 상태(ACTIVE/TRIAL 등)를 가집니다.
external_user_id자사 서비스가 관리하는 사용자 식별자. 구독 1개 규칙의 기준 키입니다.
API 키 (svc_…)자사 서버가 본인을 증명하는 아이디.
HMAC 시크릿요청 위변조를 막는 서명용 비밀키. 노출 금지.
billingKey토스에서 발급되는 자동결제용 키. 서버가 보관하며 외부에 노출되지 않습니다.
access_allowed해당 사용자에게 서비스 접근을 허용할지 판단하는 값(true/false).
단건 결제(One-off)구독과 무관한 1회성 결제.
