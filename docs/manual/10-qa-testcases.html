CHAPTER 8 · QA

QA 테스트케이스

시스템에서 발생할 수 있는 동작을 모듈별로 정리한 테스트케이스입니다. QA가 그대로 따라 실행하고 합격/불합격을 기록할 수 있도록 구성했습니다.

이 케이스들은 저장소의 자동화 테스트(약 530개)가 검증하는 동작을 사람이 따라 할 수 있게 풀어 쓴 것입니다. 즉, "이렇게 동작해야 정상"이 코드로 보장되는 항목들이라, QA의 기대 결과 기준으로 신뢰할 수 있습니다.

8.1테스트 시작 전 준비

환경

  • 결제 서버 실행(http://localhost:8000) — 설치는 6장 참고.
  • 토스 테스트 키(test_sk_…/test_ck_…) 설정 — 테스트 모드에서는 실제 카드를 넣어도 청구되지 않습니다.
  • API 테스트는 HMAC 서명이 필요합니다 — 서명 방법·예제 코드는 7. API 연동. (샘플 서비스 sample_service 8001로 화면 테스트도 가능)

테스트 데이터

  • 시스템 관리자 계정 1개, 서비스 담당자 계정 1~2개(서로 다른 서비스 담당).
  • 서비스 2개(격리 테스트용), 각 서비스에 요금제 여러 개(일반/체험/자동결제안함/할인 포함).
  • 호출 서버 IP를 서비스 허용 IP에 등록.

우선순위 표기

P1 핵심(돈·보안·접근 — 반드시 통과) · P2 중요(주요 기능) · P3 보조·엣지케이스

결과 기록

각 케이스마다 합격(Pass)/불합격(Fail)/보류(Blocked)와 실패 시 증상·스크린샷·구독ID/주문번호를 남기세요. (이 표를 엑셀로 내보내 체크리스트로 쓰고 싶으면 요청하세요.)

8.3자동 실행 & HTML 리포트

위 케이스를 손으로 다 돌리지 않아도 됩니다. 각 케이스는 저장소의 자동화 테스트(pytest)에 매핑되어 있어, 실행기를 돌리면 합격/불합격을 한 장의 HTML 리포트로 만들어 줍니다.

파일역할
docs/manual/qa/qa_cases.py140개 케이스 ↔ pytest 테스트 매핑(편집해서 유지보수)
docs/manual/qa/run_qa.pypytest 실행 → 결과 파싱 → qa-report.html 생성(표준 라이브러리만 사용)
docs/manual/qa/README.md사용법·상태 의미·유지보수 안내

저장소 루트에서 실행합니다(사전: 6장처럼 docker compose up -d + uv sync).

# ① pytest 실행 → HTML 리포트 생성
uv run python docs/manual/qa/run_qa.py
#   → docs/manual/qa/qa-report.html 를 브라우저로 열기

# ② 이미 만든 junit 결과로 리포트만
uv run pytest --junitxml=results.xml
python docs/manual/qa/run_qa.py --from-xml results.xml

# ③ DB 없이 리포트 형식 미리보기(모의 데이터)
python docs/manual/qa/run_qa.py --demo --out docs/manual/qa/qa-report.sample.html
  • 리포트에는 전체 합격률·P1 합격률·우선순위별 막대와 모듈 A~L별 케이스 표(결과 배지·매핑 테스트)가 나옵니다.
  • 불합격이 1건이라도 있으면 종료 코드 1을 반환하므로 CI에 그대로 연결할 수 있습니다.
  • 케이스를 더하거나 테스트명이 바뀌면 qa_cases.py만 수정하면 됩니다.
ℹ️

이 실행기는 tests/의 자동화 테스트를 실행만 하며 앱 코드를 변경하지 않습니다. pytest 실행에는 프로젝트 환경(Python 3.13 + DB/Redis)이 필요하고, 리포트 생성·파싱은 어떤 파이썬에서나 동작합니다.

🧰

각 케이스의 상세 동작·용어는 본문 장과 연결됩니다 — 개념 0장, 화면 5장, API 7장, 상태·에러 11장. 이 표를 엑셀 체크리스트(.xlsx)로 만들어 드릴 수도 있습니다 — 필요하면 말씀하세요.
