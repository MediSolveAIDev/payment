CHAPTER 6 · 시스템 관리자 · Docker 설치 → 배포

설치 · 설정 · 배포

서버를 처음 구축·배포하는 시스템 관리자를 위한 문서입니다. Docker로 설치·실행(nginx·app·redis 컨테이너)하는 절차부터 인스턴스 환경(사양·네트워크·구성도)과 .env 전체 레퍼런스까지 하나로 정리했습니다. DB(PostgreSQL)는 컨테이너에 넣지 않고 외부(전용 DB서버 또는 관리형 PostgreSQL)를 연결합니다.

ℹ️

이미 서버가 구축되어 있고 계정·키를 전달받은 서비스 담당자라면 이 장을 건너뛰고 0. 시작하기부터 보면 됩니다. 이 장은 처음 서버를 띄우는 사람이 한 번만 하는 작업입니다.

큰 그림은 단순합니다. ① Docker 설치 → ② .env.prod에 비밀값 채우기 → ③ TLS 인증서 두기 → ④ docker compose ... up -d 하면 컨테이너 셋(nginx·app·redis)이 뜨고, 앱이 시작할 때 DB 스키마(마이그레이션)가 자동 적용됩니다. 그다음 ⑤ 첫 관리자 계정을 만들고 ⑥ 토스를 연결하면 끝입니다.

PART 1 · 설치 · 실행
Docker로 설치하고 실행하기

Docker 설치 → 구성 파일 → 환경값(.env.prod) → 인증서 → 빌드·실행 → 첫 관리자 → 토스 연결.

1.1준비물

항목설명
클라우드 인스턴스(리눅스)앱서버 1대. 권장 4 vCPU / 8 GB (사양은 2.3). Ubuntu 22.04+ 또는 Amazon Linux 2023 기준.
Docker Engine + Compose컨테이너 실행 런타임. docker compose(v2 플러그인) 사용. 설치는 1.2.
외부 PostgreSQL전용 DB서버의 postgres 또는 관리형 PostgreSQL(Azure Database for PostgreSQL). 앱 컨테이너에서 닿는 주소·계정이 필요(DATABASE_URL).
도메인 · TLS 인증서HTTPS·토스 웹훅 수신용. Let's Encrypt 등으로 발급(테스트는 자체 서명 가능, 1.6).
토스페이먼츠 가맹점시크릿/클라이언트 키. 처음엔 테스트 키(test_sk_…)로 시작. (토스 자료: 저장소 docs/toss)
저장소 코드payment_system 프로젝트(이 매뉴얼이 든 그 저장소). 인스턴스에 git clone.

1.2Docker 설치

인스턴스에 Docker Engine과 Compose 플러그인을 설치합니다.

# Ubuntu — 공식 편의 스크립트
curl -fsSL https://get.docker.com | sudo sh

1.3저장소 구성 파일

운영 배포에 쓰는 파일들은 저장소에 포함돼 있습니다. (개발용 docker-compose.yml과 구분됩니다 — 1.10)

파일역할
Dockerfile앱 이미지 — uv로 잠금(lock) 의존성 설치 후 uvicorn 실행. Python 3.13, 비루트.
docker-compose.prod.yml운영 오케스트레이션 — app·redis·nginx 세 컨테이너.
docker/entrypoint.sh컨테이너 시작 시 alembic upgrade head(DB 스키마 적용) 후 앱 실행.
docker/nginx/conf.d/payment.confnginx TLS 리버스 프록시 — 80→443 리다이렉트, 443→app:8000.
.env.prod운영 환경값(비밀값 포함, Git 미추적). PART 3에서 전체 변수 설명.

위 파일 각각의 전체 내용과 한 줄(지시어)씩의 역할 해설은 이 장 끝의 PART 4 · 구성 파일 상세 해설에 정리했습니다. 처음엔 그대로 두고 진행해도 되지만, “이 줄이 왜 있는지”가 궁금하면 그곳을 참고하세요.

1.4한 번에 보기

처음부터 끝까지의 전체 순서입니다. 각 단계는 1.5부터 자세히 설명합니다.

git clone <저장소> payment_system && cd payment_system

cp .env.example .env.prod                      # ① 환경값 파일 만들고 값 채우기(필수값 1.5)
#  → DATABASE_URL(외부 DB)·ENCRYPTION_KEY·TOSS_SECRET_KEY·BASE_URL·SWAGGER_PW

#  ② TLS 인증서를 docker/nginx/certs/ 에 배치 (fullchain.pem, privkey.pem)

docker compose -f docker-compose.prod.yml up -d --build   # ③ 빌드+기동(시작 시 마이그레이션 자동)
docker compose -f docker-compose.prod.yml logs -f app     #    기동·마이그레이션 로그 확인

docker compose -f docker-compose.prod.yml exec app \
    python -m app.cli create-admin --email admin@회사.com --password '10자 이상 비밀번호'   # ④ 첫 관리자

#  ⑤ https://도메인/health → {"status":"ok"} · https://도메인/admin 로그인
#  ⑥ 토스 시크릿 키 연결 + 웹훅 URL 등록

1.5① 환경값 — .env.prod

.env.example을 복사해 .env.prod를 만들고 값을 채웁니다. 비밀값은 소스에 두지 말고 이 파일(또는 OS 환경변수)에만 둡니다. 처음 구축에 꼭 채워야 하는 값:

변수의미 · 채우는 법
ENCRYPTION_KEY 필수빌링키·HMAC 시크릿을 DB에 암호화 저장하는 AES-256 키(base64 32바이트). 생성:
python -c "import base64,os;print(base64.b64encode(os.urandom(32)).decode())"
TOSS_SECRET_KEY 필수토스 시크릿 키. 개발 test_sk_…, 운영 live_sk_…. (프론트 클라이언트 키와 다름 — 1.9)
DATABASE_URL 필수외부 DB 연결 문자열(asyncpg). 예: postgresql+asyncpg://app:***@10.0.0.x:5432/payments. 관리형 DB가 SSL 강제면 끝에 ?ssl=require. localhost 금지(컨테이너 자신을 가리킴 — 자주 겪는 오류).
BASE_URL공개 HTTPS 주소(메일 링크·웹훅 수신용). 예 https://api.회사도메인.com.
SWAGGER_ID · SWAGGER_PWAPI 문서(/docs) 접근용 Basic 계정. 둘 다 채우면 노출, 비우면 404(운영 권장). 예시값은 반드시 변경.
GMAIL_ID · GMAIL_PW(선택) 메일 실제 발송 시 Gmail 앱 비밀번호. 비우면 메일 내용이 로그로만 출력(개발).
🐳

compose가 자동으로 고정 주입하는 값이 있어 .env.prod에 적지 않아도 됩니다(적어도 compose 값이 우선): APP_ENV=prod, REDIS_URL=redis://redis:6379/0(컨테이너 이름), TRUST_PROXY=true, TRUST_PROXY_HOPS=1. 나머지 보안·정책 값의 의미·기본값은 PART 3 · .env 전체 레퍼런스에 있습니다.

🔑

ENCRYPTION_KEY는 절대 분실·변경하지 마세요. 이 키로 빌링키가 암호화돼 있어, 잃으면 저장된 결제수단을 복호화할 수 없습니다. 안전한 곳에 백업하세요.

1.6② TLS 인증서 배치

nginx가 HTTPS를 종단하므로 docker/nginx/certs/에 fullchain.pem·privkey.pem 두 파일이 있어야 합니다(개인키는 Git 미추적).

# 운영: Let's Encrypt (webroot = docker/nginx/certbot-www)
sudo certbot certonly --webroot -w docker/nginx/certbot-www -d api.회사도메인.com
sudo cp /etc/letsencrypt/live/api.회사도메인.com/fullchain.pem docker/nginx/certs/
sudo cp /etc/letsencrypt/live/api.회사도메인.com/privkey.pem  docker/nginx/certs/

# 테스트(도메인 없이 우선 구동): 자체 서명 인증서
openssl req -x509 -newkey rsa:2048 -nodes -days 365 \
  -keyout docker/nginx/certs/privkey.pem \
  -out   docker/nginx/certs/fullchain.pem -subj "/CN=localhost"

갱신: certbot renew 후 docker compose -f docker-compose.prod.yml exec nginx nginx -s reload.

1.7③ 빌드 · 실행 · 로그 · 업데이트

docker compose -f docker-compose.prod.yml up -d --build   # 빌드+기동(시작 시 alembic 마이그레이션 자동 적용)
docker compose -f docker-compose.prod.yml logs -f app     # 마이그레이션·기동 로그 확인
docker compose -f docker-compose.prod.yml ps              # 컨테이너 상태

docker compose -f docker-compose.prod.yml down            # 중지(redis 볼륨 유지)
git pull && docker compose -f docker-compose.prod.yml up -d --build   # 재배포

로그에 alembic upgrade head 완료 후 uvicorn 기동 메시지가 보이면 정상입니다. 외부에 열리는 포트는 nginx의 80/443뿐이며, app(8000)·redis(6379)는 도커 내부 네트워크 전용입니다.

1.8④ 첫 시스템 관리자 계정 생성 핵심

콘솔 로그인에는 계정이 필요한데, 맨 처음 관리자 계정은 명령으로 직접 만듭니다(이후 계정은 콘솔 화면에서 추가). 앱 컨테이너 안에서 실행합니다.

docker compose -f docker-compose.prod.yml exec app \
    python -m app.cli create-admin \
    --email admin@회사.com \
    --password '10자 이상 비밀번호'

성공하면 SYSTEM_ADMIN 생성 완료: admin@회사.com이 출력됩니다. 이 계정으로 로그인합니다.

1.9⑤ 첫 로그인 · ⑥ 토스 연결

  1. 동작 확인

    https://도메인/health → {"status":"ok"}가 보이면 정상.

  2. 콘솔 로그인

    https://도메인/admin에서 1.8의 관리자 계정으로 로그인 → 대시보드가 보이면 성공. 로그인 화면 하단의 “📚 전체 매뉴얼 보기” 링크로 이 매뉴얼을 바로 열 수 있습니다(로그인 불필요).

📚

이 매뉴얼은 서버에서 https://도메인/manual/ 로도 서빙됩니다(app/main.py의 /manual 정적 마운트). 로그인 페이지에서 인증 없이 링크돼 있어, 담당자가 키를 받기 전에도 열람할 수 있습니다. 운영 이미지에는 .dockerignore 설정으로 docs/manual만 포함됩니다.

실제 결제가 되려면 토스 키 2종을 각자 다른 위치에 둡니다.

키두는 곳용도
시크릿 키 (…sk_…)결제 서버 .env.prod → TOSS_SECRET_KEY서버→토스 결제 승인·빌링키 발급(서버 간)
클라이언트 키 (…ck_…)각 서비스의 프론트엔드브라우저에서 토스 카드 등록창(SDK)

웹훅(Webhook) 등록

  • 수신 URL — (BASE_URL)/api/v1/webhooks/toss (예: https://api.회사도메인.com/api/v1/webhooks/toss)
  • 허용 IP — 토스 공식 발신 IP만 받습니다(WEBHOOK_IP_CHECK_ENABLED=true, 목록은 TOSS_WEBHOOK_ALLOWED_IPS에 기본 포함). 개발 중 검증을 끄려면 false.
⚠️

웹훅은 외부(토스)에서 들어오므로 BASE_URL이 인터넷에서 접근 가능한 HTTPS 주소여야 합니다. 토스 대시보드 등록 위치·테스트 방법은 저장소 docs/toss 참고.

1.10(참고) 로컬 개발 실행

운영 배포가 아니라 로컬에서 빠르게 개발할 때는 개발용 docker-compose.yml로 DB·Redis만 띄우고 앱은 uv로 직접 실행합니다.

docker compose up -d                  # 개발용 PostgreSQL(5433)·Redis(6380)
cp .env.example .env.dev              # 개발 환경값
uv sync                               # 의존성 설치
uv run alembic upgrade head           # 스키마 생성
uv run python -m app.cli create-admin --email admin@회사.com --password '10자 이상'
uv run uvicorn app.main:app --reload  # http://localhost:8000/admin
⛔

개발용 docker-compose.yml은 비밀번호가 payment로 고정된 개발 전용입니다. 운영에서는 절대 그대로 쓰지 말고, 위 docker-compose.prod.yml + 외부 DB + 강한 자격증명을 사용하세요.

1.11구축 완료 체크리스트

  • 인스턴스에 Docker·Compose 설치 확인(docker compose version).
  • .env.prod에 DATABASE_URL(외부 DB)·ENCRYPTION_KEY·TOSS_SECRET_KEY·BASE_URL를 채우고 SWAGGER_PW 변경.
  • docker/nginx/certs/에 TLS 인증서 2개 배치.
  • up -d --build 후 logs -f app에서 마이그레이션·기동 정상.
  • create-admin으로 첫 관리자 계정 생성.
  • /health가 ok, /admin 로그인 성공.
  • 토스 시크릿 키(서버)·클라이언트 키(서비스 프론트) 배치 + 웹훅 URL 등록.
  • ENCRYPTION_KEY를 안전하게 백업, DB 자동 백업 설정.
PART 2 · 인스턴스 환경
배포 아키텍처 · 인스턴스 사양 · 네트워크

앱서버(nginx+app+redis 컨테이너) + 외부 DB(전용 서버 또는 관리형 PostgreSQL) 구성.

2.1구성 개요

앱서버 1대에 nginx · app · redis를 docker로 띄우고, DB(PostgreSQL)는 분리합니다 — 전용 DB서버의 postgres이거나 관리형 PostgreSQL입니다. 돈 기록(DB)을 앱과 분리해 백업·보안을 단순하게 만드는 것이 핵심입니다.

비유하면 — app은 주문받는 점원, PostgreSQL은 금고·장부(돈 기록, 절대 분실 금지 → 백업), redis는 포스트잇 메모(로그인·잠금 같은 임시 정보), nginx는 입구 안내데스크(HTTPS). 금고(DB)만 안전한 곳에 따로 두고, 나머지는 한 박스(앱서버)에 모았다고 보면 됩니다.

인스턴스 ↔ docker 매핑

대상역할구성외부 노출
① 앱서버입구 · 애플리케이션 · 공유 캐시nginx + app + redis (docker)HTTPS 443·80만 공개
② DB영구 데이터(돈 기록)전용 DB서버의 postgresql 또는 Azure Database for PostgreSQL비공개 — 사설망/보안그룹에서 앱서버만 접근

구성도

인스턴스당 권장 사양

사내 공용 · 중소 트래픽(피크 수십~수백 RPS 이하) 기준 권장값입니다(정확한 값은 부하 테스트로 확정).

대상구성권장최소디스크
① 앱서버nginx+app+redis4 vCPU / 8 GB2 vCPU / 4 GB50 GB SSD (OS·이미지·로그)
② DBpostgresql / 관리형4 vCPU / 16 GB2 vCPU / 8 GB데이터 전용 100 GB SSD(IOPS 보장)

예시 타입(Azure 기준) — 앱서버(4 vCPU/8 GB): Standard_F4s_v2(4/8)·여유 Standard_D4s_v5(4/16) / 최소 Standard_D2s_v5(2/8) · DB(4 vCPU/16 GB): Azure Database for PostgreSQL — 유연한 서버 General Purpose D4ds_v5.

사양 근거

  • 앱서버 8 GB 권장 — app(uvicorn 워커)·redis·nginx가 RAM을 나눠 씁니다. redis maxmemory를 1~2 GB로 못 박아 OOM을 막으세요.
  • DB는 메모리가 성능 — PostgreSQL 캐시(shared_buffers ≈ RAM 25%)가 RAM에 올라가야 빠릅니다. 가장 중요한 한 대라 16 GB 권장.
  • DB 데이터는 OS와 분리된 SSD — IOPS 보장 볼륨 + 자동 스냅샷. 결제 부하가 늘면 이 디스크가 먼저 병목.

네트워크 · 보안 원칙

  • 앱은 무상태(stateless) — 세션·락·요청제한은 redis에, 영구 데이터는 전부 DB에. 앱 컨테이너는 언제든 재시작·교체 가능.
  • 포트 최소 노출 — 앱서버는 80/443만 공개. app(8000)·redis(6379)는 docker 내부 네트워크 전용. DB 5432는 앱서버만 접근(인터넷 비공개).
  • nginx가 HTTPS 입구 — 앱(8000)을 공인 443에 직접 노출하지 말고 nginx 뒤에 둡니다(TLS 종단).
  • 클라이언트 IP 판별 — nginx가 X-Forwarded-For를 세팅하고 앱은 TRUST_PROXY=true·TRUST_PROXY_HOPS=1로 읽습니다(서비스/어드민 IP 화이트리스트 정확도에 직결). 앞단에 LB가 하나 더 있으면 TRUST_PROXY_HOPS=2.
  • redis 보호 — 내부 네트워크 전용 + 필요 시 requirepass. maxmemory 한도·제거정책 설정.
  • 시크릿 백업 — ENCRYPTION_KEY 분실 시 빌링키 복호화 불가.

운영 핵심

  • DB 백업은 필수·자동 — 관리형이면 자동 백업·스냅샷을 켜고, 직접 운영이면 pg_dump/스냅샷을 매일 떠 인스턴스 밖(오브젝트 스토리지 등)에 보관. 결제 데이터엔 타협 불가.
  • 스케줄러 — 앱이 1대이므로 SCHEDULER_ENABLED=true면 자동결제 갱신 배치가 앱서버에서 돕니다(Redis 분산 락이 중복 실행을 차단).
  • 마이그레이션 — 컨테이너 시작 시 자동 적용. 앱을 2대 이상으로 늘릴 땐 한 대만 적용하도록 나머지는 RUN_MIGRATIONS=0으로 띄웁니다.
  • 무중단은 아님 — 앱이 1대라 재배포 시 짧은 중단이 생깁니다. 점검 공지 또는 트래픽 적은 시간대 배포.
🔁

앱을 늘린다면 — 앱서버를 2대 이상으로 확장할 땐 redis를 앱 밖의 공유 위치로 옮겨 모든 앱이 같은 redis를 보게 해야 합니다(REDIS_URL을 그 주소로). 안 그러면 락·세션이 분리돼 자동결제 중복·세션 꼬임이 생깁니다.

⚠️

자주 겪는 오류

  • DB 접속 실패 Connect call failed ('127.0.0.1', 5433) — DATABASE_URL이 localhost이면 컨테이너 자신을 가리켜 실패합니다. 외부 DB는 사설 IP·관리형 DB 엔드포인트로, 호스트의 개발 DB로 테스트할 땐 host.docker.internal로 지정(관리형 DB가 SSL 강제면 ?ssl=require).
  • nginx 기동 실패 cannot load certificate ".../fullchain.pem" — 인증서 파일이 없습니다. 1.6으로 먼저 배치하세요.
PART 3 · 설정
.env 전체 레퍼런스

로딩 순서 · 전체 환경변수와 기본값 · 런타임 변경 가능 항목.

3.1.env 로딩 순서

공통 .env를 먼저 읽고 환경별 .env.<APP_ENV>(예: .env.dev/.env.prod)로 덮어씁니다(뒤가 우선). 환경 선택은 OS 변수 APP_ENV(없으면 ENVIRONMENT, 기본 dev)로 합니다.

  • 개발: uv run uvicorn … → .env + .env.dev
  • 운영(직접 실행): APP_ENV=prod uvicorn … → .env + .env.prod
  • 운영(Docker): compose가 .env+.env.prod를 컨테이너 환경변수로 주입. OS/컨테이너 환경변수가 .env 파일보다 우선하므로 compose가 APP_ENV=prod·REDIS_URL·TRUST_PROXY를 고정합니다.
ℹ️

선언되지 않은 변수는 무시됩니다(extra="ignore"). 값은 프로세스 시작 시 한 번 읽으므로 변경 후 재시작이 필요합니다(런타임 표시 값은 예외 — 콘솔 전체 설정에서 재배포 없이 즉시 변경, DB 전역설정이 우선).

PART 4 · 구성 파일 상세 해설
Docker 관련 파일 — 전체 내용과 줄별 역할

저장소에 든 Docker 파일을 실제 내용 그대로 싣고, 각 줄(지시어)이 어떤 역할을 하는지 표로 풀어 설명합니다. 값을 바꿔야 할 때 “이 줄을 건드려도 되는지”를 판단하는 근거로 쓰세요.

🗂️

파일 4개는 서로 맞물려 한 흐름을 만듭니다 — Dockerfile(앱 이미지를 어떻게 만들지) → docker-compose.prod.yml(그 이미지를 redis·nginx와 함께 어떻게 띄울지) → entrypoint.sh(컨테이너가 켜질 때 DB 스키마부터 맞춤) → nginx/payment.conf(바깥 HTTPS 요청을 앱으로 전달). .dockerignore는 이미지를 가볍고 안전하게 굽는 보조 파일입니다.

4.1Dockerfile — 앱 이미지 굽기

앱(FastAPI) 컨테이너 이미지를 만드는 설계도입니다. uv로 잠금된 의존성을 설치하고 uvicorn으로 실행합니다. DB는 이미지에 넣지 않고 외부를 연결합니다.

# payment_system(FastAPI) 운영 이미지 — uv로 잠금(lock)된 의존성 설치 후 uvicorn 실행.
# DB(PostgreSQL)는 외부 관리형(RDS 등)을 DATABASE_URL로 연결한다(이미지에 미포함).
FROM python:3.13-slim

# 파이썬 런타임 위생 + uv 설정
ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1 \
    UV_COMPILE_BYTECODE=1 \
    UV_LINK_MODE=copy \
    UV_PROJECT_ENVIRONMENT=/app/.venv \
    PATH="/app/.venv/bin:$PATH"

WORKDIR /app

# uv 설치(공식 정적 바이너리 복사 — pip보다 빠르고 캐시 친화적)
COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv

# 1) 의존성만 먼저 설치 — 소스가 바뀌어도 이 레이어는 캐시됨
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev --no-install-project

# 2) 앱 소스 + 마이그레이션
COPY app ./app
COPY alembic ./alembic
COPY alembic.ini ./
COPY docker/entrypoint.sh /usr/local/bin/entrypoint.sh
RUN chmod +x /usr/local/bin/entrypoint.sh

# 비루트 실행
RUN useradd --create-home --uid 10001 appuser && chown -R appuser:appuser /app
USER appuser

EXPOSE 8000
# 엔트리포인트가 alembic 마이그레이션을 적용한 뒤 CMD를 exec 한다.
ENTRYPOINT ["entrypoint.sh"]
# 앱 포트(8000)는 외부에 공개하지 않고 nginx만 접근하므로 프록시 헤더를 신뢰한다.
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", \
     "--proxy-headers", "--forwarded-allow-ips", "*"]
지시어역할
FROM python:3.13-slim베이스 이미지. 데비안 슬림 + Python 3.13. 군더더기 없이 작아 이미지·공격면이 작습니다.
ENV PYTHONUNBUFFERED=1표준출력 버퍼링을 끔 → 앱 로그가 지연 없이 즉시 docker logs에 보입니다.
PYTHONDONTWRITEBYTECODE=1실행 중 .pyc 캐시 파일을 만들지 않음(컨테이너엔 불필요한 군더더기).
UV_COMPILE_BYTECODE=1반대로 설치 시점에 바이트코드를 미리 컴파일 → 첫 기동이 빨라집니다.
UV_LINK_MODE=copyuv가 패키지를 하드링크 대신 복사로 배치(레이어·마운트 경계에서 안전).
UV_PROJECT_ENVIRONMENT=/app/.venv가상환경 위치를 /app/.venv로 고정.
PATH="/app/.venv/bin:$PATH"venv의 실행파일(uvicorn·alembic 등)을 경로에 올려 바로 호출 가능하게.
WORKDIR /app이후 명령의 기준 작업 디렉터리.
COPY --from=…/uv …uv 정적 바이너리를 공식 이미지에서 복사. pip보다 빠른 의존성 설치 도구.
COPY pyproject.toml uv.lock + RUN uv sync …레이어 캐시 전략. 의존성 파일만 먼저 복사·설치하므로 앱 코드만 바뀌면 이 무거운 레이어는 재사용돼 빌드가 빠릅니다.
--frozenuv.lock을 그대로 설치(재해석·갱신 금지) → 빌드 재현성 보장.
--no-dev테스트·린트 같은 개발 전용 의존성 제외(운영 이미지 슬림화).
--no-install-project이 단계에선 앱 코드는 아직 설치 안 함(의존성 레이어만 분리하려는 것).
COPY app / alembic / alembic.ini앱 소스와 DB 마이그레이션 파일을 이미지에 복사.
COPY docker/entrypoint.sh … + chmod +x시작 스크립트를 설치하고 실행 권한 부여(4.3 참조).
useradd … appuser + USER appuser비루트(uid 10001)로 실행 → 컨테이너가 탈취돼도 루트 권한 피해를 막습니다(보안 기본).
EXPOSE 8000앱이 듣는 포트를 문서로 선언. 실제 외부 공개 여부는 compose가 결정(여기선 미공개).
ENTRYPOINT ["entrypoint.sh"]컨테이너가 켜지면 항상 엔트리포인트 먼저 실행(= DB 마이그레이션 선행).
CMD ["uvicorn", …]엔트리포인트가 마지막에 exec할 실제 앱 실행 명령. --proxy-headers·--forwarded-allow-ips * = nginx가 붙여 준 X-Forwarded-* 헤더를 신뢰(클라이언트 IP 판별).

4.2docker-compose.prod.yml — 세 컨테이너 오케스트레이션

위 이미지를 redis·nginx와 함께 어떻게 묶어 띄울지 정의합니다. DB는 여기에 없습니다(외부 연결).

services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
    # .env(공통) + .env.prod(운영) 값을 컨테이너 환경변수로 주입.
    env_file:
      - .env
      - .env.prod
    environment:
      APP_ENV: prod
      # 인프라 결선 — .env 값과 무관하게 컨테이너 네트워크 기준으로 고정
      REDIS_URL: redis://redis:6379/0
      # nginx 뒤에 있으므로 X-Forwarded-For 신뢰
      TRUST_PROXY: "true"
      TRUST_PROXY_HOPS: "1"
    depends_on:
      redis:
        condition: service_healthy
    # 로컬 테스트 시 호스트의 개발용 Postgres에 닿게 함(실제 배포에선 미사용·무해)
    extra_hosts:
      - "host.docker.internal:host-gateway"
    expose:
      - "8000"          # 외부로 직접 노출하지 않음 — nginx만 접근
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "python", "-c", "import socket,sys; s=socket.create_connection(('127.0.0.1',8000),3); s.close()"]
      interval: 15s
      timeout: 5s
      retries: 5
      start_period: 30s

  redis:
    image: redis:7-alpine
    # 세션·nonce·레이트리밋·킬스위치 캐시 저장소. 내부 네트워크에서만 접근(포트 미공개).
    command: ["redis-server", "--save", "60", "1", "--appendonly", "no"]
    volumes:
      - redis-data:/data
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5

  nginx:
    image: nginx:1.27-alpine
    depends_on:
      - app
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./docker/nginx/conf.d:/etc/nginx/conf.d:ro
      - ./docker/nginx/certs:/etc/nginx/certs:ro
      - ./docker/nginx/certbot-www:/var/www/certbot   # Let's Encrypt webroot(갱신용)
    restart: unless-stopped

volumes:
  redis-data:
키역할
app.build.context/dockerfile현재 폴더(.)를 빌드 컨텍스트로 Dockerfile을 빌드해 app 이미지를 만듭니다.
app.env_file: .env, .env.prod두 파일의 값을 컨테이너 환경변수로 주입(뒤가 우선 → 운영값이 공통값을 덮음).
environment.APP_ENV: prod실행 환경을 운영으로 강제(.env 로딩 분기·개발 편의 기능 차단).
REDIS_URL: redis://redis:6379/0같은 compose 네트워크의 컨테이너 이름 redis로 결선. 호스트 포트와 무관하게 고정.
TRUST_PROXY / TRUST_PROXY_HOPSnginx 뒤에 있으므로 X-Forwarded-For를 신뢰하고 오른쪽 1번째를 클라이언트 IP로 사용(IP 화이트리스트 정합).
depends_on.redis: service_healthyredis의 헬스체크가 통과한 뒤에 app을 시작(캐시 미준비 상태 기동 방지).
extra_hosts: host.docker.internal로컬 테스트에서 호스트의 개발 DB에 닿기 위한 별칭(리눅스 포함). 실제 외부 DB 배포에선 안 쓰여 무해.
expose: "8000"포트를 내부 네트워크에만 노출(호스트로 publish 아님) → nginx만 접근.
restart: unless-stopped컨테이너가 죽으면 자동 재시작(사람이 down으로 멈춘 경우만 제외).
app.healthcheck8000 TCP 연결로 앱 생존 확인. start_period 30s는 기동·마이그레이션 동안 실패로 안 세는 유예.
redis.image: redis:7-alpine경량 Redis 7. 세션·nonce·레이트리밋·킬스위치 캐시 저장.
redis.command --save 60 1 --appendonly no60초 내 키 1개 이상 변경 시 스냅샷 저장, AOF 로그는 끔. 캐시·세션 용도라 강한 영속성이 불필요.
redis.volumes redis-data:/data스냅샷을 명명 볼륨에 보관 → 재시작해도 세션이 유지.
nginx.ports "80:80" "443:443"유일하게 외부(호스트)로 공개되는 포트. 모든 인터넷 트래픽의 입구.
nginx.volumes …:ro설정(conf.d)·인증서(certs)를 읽기전용(ro)으로 마운트, certbot-www는 인증서 갱신용 webroot.
volumes: redis-data위에서 쓴 명명 볼륨을 선언(데이터 보관 영역).

4.3docker/entrypoint.sh — 시작 시 DB 스키마 자동 적용

컨테이너가 켜질 때 가장 먼저 실행됩니다. DB 스키마를 최신으로 맞춘 뒤 앱(CMD)을 넘겨받아 실행합니다.

#!/bin/sh
# 컨테이너 시작 시 DB 스키마를 최신으로 맞춘 뒤(alembic) 앱(CMD)을 실행한다.
set -e

if [ "${RUN_MIGRATIONS:-1}" != "0" ]; then
  echo "[entrypoint] applying DB migrations (alembic upgrade head)..."
  alembic upgrade head
else
  echo "[entrypoint] RUN_MIGRATIONS=0 — skipping migrations"
fi

echo "[entrypoint] starting: $*"
exec "$@"
줄역할
#!/bin/shPOSIX sh로 실행(alpine·slim에 기본 포함되는 셸).
set -e어느 명령이라도 실패하면 즉시 중단 → 마이그레이션이 실패했는데 앱이 뜨는 상황을 막습니다.
if RUN_MIGRATIONS != 0 … alembic upgrade head기본값으로 DB 스키마를 최신(head)까지 적용. DATABASE_URL 환경변수가 대상 DB를 가리킵니다.
RUN_MIGRATIONS=0 분기앱을 여러 대로 늘릴 때 한 대만 마이그레이션하고 나머지는 건너뛰게 하는 안전장치(중복 적용 방지).
exec "$@"전달받은 CMD(uvicorn)를 PID 1로 교체 실행. 셸이 중간에 끼지 않아 종료 시그널(SIGTERM)이 앱에 정확히 전달돼 깔끔하게 종료됩니다.

4.4docker/nginx/conf.d/payment.conf — TLS 리버스 프록시

인터넷의 HTTPS 요청을 받아 TLS를 종단하고 내부 app:8000으로 전달합니다. 80은 모두 https로 리다이렉트합니다.

upstream payment_app {
    server app:8000;
    keepalive 32;
}

# ── HTTP(80): ACME(Let's Encrypt) 챌린지 외에는 전부 https로 ──
server {
    listen 80;
    server_name _;
    location /.well-known/acme-challenge/ {   # 인증서 발급/갱신용
        root /var/www/certbot;
    }
    location / {
        return 301 https://$host$request_uri;
    }
}

# ── HTTPS(443): 실제 서비스 ──
server {
    listen 443 ssl;
    http2 on;
    server_name _;

    ssl_certificate     /etc/nginx/certs/fullchain.pem;
    ssl_certificate_key /etc/nginx/certs/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 1h;

    client_max_body_size 16m;   # 엑셀 업로드/다운로드 등

    add_header X-Content-Type-Options nosniff always;
    add_header X-Frame-Options DENY always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    location / {
        proxy_pass http://payment_app;
        proxy_http_version 1.1;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Connection        "";
        proxy_read_timeout 90s;   # 토스 자동결제 승인(최대 65s)보다 넉넉히
    }
}
지시어역할
upstream payment_app { server app:8000; keepalive 32; }백엔드 풀 정의. app 컨테이너의 8000으로 보내고, 연결 32개를 재사용(keepalive)해 매 요청 TCP 핸드셰이크 비용을 줄입니다.
listen 80 / location /.well-known/acme-challenge/80 포트는 Let's Encrypt 인증서 발급·갱신 경로만 webroot로 처리.
location / { return 301 https://… }나머지 모든 80 요청을 HTTPS로 영구 리다이렉트(평문 차단).
listen 443 ssl + http2 on실제 서비스 포트. TLS + HTTP/2 활성화.
ssl_certificate / ssl_certificate_key마운트된 인증서·개인키 경로(1.6에서 배치). 없으면 nginx가 기동 실패.
ssl_protocols TLSv1.2 TLSv1.3 / ssl_ciphers HIGH:…구버전 취약 프로토콜·암호 제외, 안전한 조합만 허용.
ssl_session_cache / timeoutTLS 세션을 캐시해 재접속 핸드셰이크를 생략(성능).
client_max_body_size 16m업로드 본문 최대 크기(엑셀 업로드 등). 더 큰 파일이 필요하면 상향.
add_header … nosniff / DENY / HSTS보안 헤더 보강 — MIME 스니핑 차단·클릭재킹(iframe) 차단·HTTPS 강제(HSTS).
proxy_pass http://payment_app요청을 위 upstream(앱)으로 전달.
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for클라이언트 IP를 XFF 맨 오른쪽에 append → 앱의 TRUST_PROXY_HOPS=1과 정확히 맞물려 진짜 IP를 식별.
X-Real-IP / X-Forwarded-Proto원 클라이언트 IP와 원 프로토콜(https)을 앱에 전달.
proxy_set_header Connection ""업스트림 keepalive를 살리기 위해 hop-by-hop Connection 헤더 제거.
proxy_read_timeout 90s응답 대기 한도. 토스 자동결제 승인(최대 65초)이 끊기지 않도록 넉넉히 잡음.

4.6(참고) docker-compose.yml — 개발 전용 인프라

운영 파일과 헷갈리지 않도록 정리합니다. 이건 로컬 개발에서 Postgres·Redis만 띄우는 용도이며(앱은 uv로 직접 실행 — 1.10), 운영에 쓰면 안 됩니다.

# 개발 전용 인프라(Postgres/Redis). 운영에서는 이 파일을 그대로 쓰지 말 것.
# 포트는 루프백(127.0.0.1)에만 바인딩한다(보안: 외부 노출 방지).
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: payment
      POSTGRES_PASSWORD: payment   # 개발 전용 — 운영은 별도 자격증명 필수
      POSTGRES_DB: payment
    ports:
      - "127.0.0.1:5433:5432"
    volumes:
      - ./scripts/init-db.sql:/docker-entrypoint-initdb.d/init-db.sql
      - pgdata:/var/lib/postgresql/data
  redis:
    image: redis:7-alpine
    ports:
      - "127.0.0.1:6380:6379"
volumes:
  pgdata:
항목역할
postgres:16-alpine + POSTGRES_*개발용 로컬 DB. 계정·비번·DB명을 payment로 고정(개발 전용).
ports "127.0.0.1:5433:5432"루프백에만 바인딩 — Docker 기본(0.0.0.0)으로 띄우면 외부에서 DB가 노출되므로, 호스트 안에서만 접근하도록 제한(호스트 포트 5433).
volumes init-db.sql · pgdata최초 1회 초기화 SQL 실행 + 데이터 영속 볼륨.
redis "127.0.0.1:6380:6379"개발용 Redis도 루프백 6380에만 노출.
