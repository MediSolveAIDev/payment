사전 개발 — API 계약 & 덤프 데이터

결제 서버가 아직 오픈되지 않았어도, 서비스 개발자가 구독·결제·내역 조회 화면과 로직을 미리 끝까지 개발할 수 있도록 실제와 동일한 응답 스키마와 덤프 데이터(JSON)를 제공합니다.

핵심은 "가짜 응답으로 먼저 만든다"입니다. 아래 덤프 JSON은 실제 서버가 돌려줄 응답과 형식이 똑같습니다. 이걸 우리 코드의 가짜 응답(fixture/stub)으로 끼워 넣으면 서버 없이도 화면과 로직을 완성할 수 있고, 나중에 서버가 열리면 베이스 URL과 키만 바꾸면 됩니다.

📁

덤프 파일 위치: docs/manual/dev-sandbox/dump/ · 안내: 같은 폴더 README.md. API의 정식 인증·서명·코드 예제는 7. API 연동, 상태·에러 의미는 11. FAQ을 참고하세요.

9.0처음이라면 — 5분 스타터

한 문장 흐름: 사용자가 요금제를 고르고 → (실서버에선) 토스 카드창에서 카드를 등록하면 auth_key가 나오고 → 내 백엔드가 POST /subscriptions를 호출하면 구독이 시작되고 → 이후엔 access_allowed 값으로 기능을 잠그거나 풀어 줍니다. 사전 개발에선 토스 카드창을 건너뛰고(auth_key는 아무 값), 나머지를 덤프로 먼저 만듭니다. (그림: 한눈에 보기 C)

용어 한 줄 사전

  • 덤프(dump) / 픽스처(fixture) — 실제 서버가 줄 응답을 미리 저장해 둔 JSON 파일.
  • 스텁(stub) — 실제 호출 함수를 "정해진 값만 돌려주는 가짜"로 바꿔치기.
  • 목(mock) / MSW — 네트워크 요청을 가로채 가짜 응답을 주는 것. MSW는 프런트(브라우저/노드)용 대표 도구.
⚠️

덤프는 "고정 응답"입니다. 무엇을 보내든 항상 같은 JSON을 돌려줍니다. 그래서 구독을 만든 뒤 조회해도 내가 보낸 값이 반영되진 않습니다. 다른 상태를 보고 싶으면 응답으로 쓰는 덤프 파일을 바꾸면 됩니다(예: subscription.get.json → subscription.suspended.json).

스타터 ① 프런트엔드 (MSW) — 복사해서 그대로

# 1) 설치 + 덤프 복사
npm i -D msw
#   docs/manual/dev-sandbox/dump/*.json  →  내 프로젝트 src/mocks/dump/ 로 복사

# 2) src/mocks/handlers.js
import { http, HttpResponse } from 'msw'
import plans     from './dump/plans.json'
import subCreate from './dump/subscription.create.json'
import subActive from './dump/subscription.get.json'
import payments  from './dump/payments.list.json'
export const handlers = [
  http.get ('*/api/v1/plans',                () => HttpResponse.json(plans)),
  http.post('*/api/v1/subscriptions',        () => HttpResponse.json(subCreate, { status: 201 })),
  http.get ('*/api/v1/subscriptions/:userId',() => HttpResponse.json(subActive)),
  http.get ('*/api/v1/payments/:userId',     () => HttpResponse.json(payments)),
]

# 3) src/mocks/browser.js
import { setupWorker } from 'msw/browser'
import { handlers } from './handlers'
export const worker = setupWorker(...handlers)

# 4) 앱 시작 전에 켜기 (예: src/main.js 맨 위)
if (import.meta.env.DEV) {
  const { worker } = await import('./mocks/browser')
  await worker.start({ onUnhandledRequest: 'bypass' })
}

# 5) 서비스워커 파일 생성(최초 1회)
npx msw init public/ --save

#  → 이제 화면 코드의 fetch('/api/v1/plans') 가 plans.json 으로 응답합니다.

스타터 ② 백엔드 (Python) — 덤프 클라이언트로 교체

실제 호출 모듈(예: payment_client) 대신 아래 파일을 import 하면 끝입니다. 에러도 흉내 냅니다.

# mocks/dump/ 에 덤프 14개를 복사한 뒤, mock_payment_client.py 를 import
import json, pathlib
DUMP = pathlib.Path(__file__).parent / "mocks" / "dump"
def _load(n): return json.loads((DUMP / n).read_text(encoding="utf-8"))

class PaymentAPIError(Exception):
    def __init__(self, status, code, message):
        super().__init__(message); self.status, self.code, self.message = status, code, message

def get_plans(creds=None):              return _load("plans.json")["plans"]
def get_subscription(uid, creds=None):  return _load("subscription.get.json")  # 상태 바꾸려면 파일명만 교체
def get_payments(uid, creds=None):      return _load("payments.list.json")["payments"]

def create_subscription(*, plan_id, external_user_id, trial=False, **kw):
    # 체험 미지원 요금제에 trial 요청 → 에러 흉내
    if trial and not plan_id.startswith("3333"):
        raise PaymentAPIError(422, "VALIDATION_ERROR", "이 요금제는 체험을 제공하지 않습니다")
    return _load("subscription.create.json")

def create_one_off_payment(*, amount, **kw):
    if amount <= 0:
        raise PaymentAPIError(422, "VALIDATION_ERROR", "amount 는 1원 이상")
    return _load("payment.oneoff.create.json")

def cancel_one_off_payment(order_id, **kw):  return _load("payment.oneoff.cancel.json")
#  → 나중에 실제 payment_client 로 import 만 되돌리면 그대로 동작합니다.

더 많은 방식(responses/respx, 정적 서빙, Postman)은 9.3 도구·스택별 사용법에 있습니다.

9.2개발 워크플로 — 화면 순서대로 따라가기

실제 서비스가 결제 서버를 쓰는 순서 그대로, 각 단계에서 어떤 덤프를 쓰면 되는지입니다.

  1. 요금제 화면

    GET /plans 자리에 plans.json을 끼웁니다. 첫 결제액·정기 결제액·체험·자동결제 배지를 화면에 그립니다.

  2. 카드 등록(건너뛰기)

    실제로는 토스 카드창에서 auth_key를 받지만, 사전 개발에선 아무 문자열로 둡니다(예: "dev-auth"). 카드 UI는 토스 SDK 연동 단계(7.11)에서 붙입니다.

  3. 구독 시작

    POST /subscriptions 자리에 subscription.create.json(201)을 반환. access_allowed로 이용 허용을 결정하는 로직을 만듭니다.

  4. 내 구독 화면

    GET /subscriptions/{id}에 상태별 덤프 subscription.get.json(ACTIVE), subscription.trial.json, subscription.past_due.json, subscription.suspended.json, subscription.canceled.json, subscription.expired.json 를 번갈아 끼워 모든 상태의 화면을 만듭니다(9.6).

  5. 결제 내역 화면

    GET /payments/{id}에 payments.list.json(성공·실패·취소 혼합)과 payments.empty.json(빈 내역)을 끼워 두 경우를 모두 처리합니다.

  6. 단건 결제 / 취소

    payment.oneoff.create.json → 결제 결과, payment.oneoff.cancel.json → 취소 후(수수료·환불) 화면을 만듭니다.

  7. 에러 처리

    errors.json의 각 코드(409 중복, 422 검증 등)를 반환해 분기·메시지를 미리 검증합니다(9.6).

9.3도구·스택별 사용법

덤프 JSON을 HTTP 응답 대신 끼우는 방법은 여러 가지입니다. 팀 환경에 맞는 하나를 고르세요.

A. 코드에서 직접 stub (백엔드)

클라이언트의 HTTP 호출부만 잠깐 바꿔 끼웁니다. 가장 빠르게 시작할 수 있습니다.

# Python — payment_client 를 덤프 버전으로 교체
import json, pathlib
DUMP = pathlib.Path("docs/manual/dev-sandbox/dump")
def _load(name): return json.loads((DUMP / name).read_text(encoding="utf-8"))

def get_plans(creds=None):                       # 실제: GET /api/v1/plans
    return _load("plans.json")["plans"]

def create_subscription(**kwargs):               # 실제: POST /api/v1/subscriptions
    return _load("subscription.create.json")

def get_subscription(uid, creds=None):           # 실제: GET /api/v1/subscriptions/{id}
    return _load("subscription.get.json")        # 상태별로 파일만 바꿔 테스트

def get_payments(uid, creds=None):               # 실제: GET /api/v1/payments/{id}
    return _load("payments.list.json")["payments"]

B. responses / respx 로 HTTP 가로채기 (Python 테스트)

실제 호출 코드를 그대로 두고, HTTP 레이어만 가짜 응답으로 응답하게 합니다.

import json, responses, requests

@responses.activate
def test_subscribe_flow():
    base = "http://localhost:8000/api/v1"
    responses.get(f"{base}/plans", json=json.load(open("dump/plans.json")))
    responses.add(responses.POST, f"{base}/subscriptions",
                  json=json.load(open("dump/subscription.create.json")), status=201)
    # ↓ 실제 우리 서비스 코드 호출 — 네트워크 없이 덤프가 응답
    plans = requests.get(f"{base}/plans").json()["plans"]
    sub = requests.post(f"{base}/subscriptions", json={...}).json()
    assert sub["access_allowed"] is True

C. MSW 로 가로채기 (프런트엔드 · React/Vue 등)

브라우저/노드의 fetch·XHR을 가로채 덤프를 돌려줍니다. 화면 개발에 가장 편합니다.

// handlers.js  (Mock Service Worker)
import { http, HttpResponse } from 'msw'
import plans      from './dump/plans.json'

  http.get('*/api/v1/plans',                 () => HttpResponse.json(plans)),
  http.post('*/api/v1/subscriptions',        () => HttpResponse.json(subCreate, { status: 201 })),
  http.get('*/api/v1/subscriptions/:userId', () => HttpResponse.json(subActive)),
  http.get('*/api/v1/payments/:userId',      () => HttpResponse.json(payments)),
  // 에러 케이스: 중복 구독 409
  // http.post('*/api/v1/subscriptions', () =>
  //   HttpResponse.json({ error:{ code:'CONFLICT', message:'이미 구독 중' }}, { status:409 })),
]

D. 정적 파일 서빙 / Postman Mock (서버 코드 없이 GET 테스트)

GET 응답만 빠르게 띄우려면 덤프 폴더를 정적 서버로 올립니다(POST는 위 A~C 또는 목 도구 사용).

# 덤프 폴더를 그대로 HTTP로 노출 (GET 전용)
cd docs/manual/dev-sandbox/dump
python3 -m http.server 8000
#  → GET http://localhost:8000/plans.json          (요금제)
#  → GET http://localhost:8000/payments.list.json  (결제 내역)
  • Postman Mock Server — 각 요청의 Example 응답 본문에 해당 덤프 JSON을 붙여 등록하면 GET·POST 모두 흉내 낼 수 있습니다.
  • WireMock / json-server — 라우트별로 덤프를 매핑해 실제 URL 형태(/api/v1/...)로 응답하게 할 수 있습니다.

에러 응답 흉내내기

성공뿐 아니라 실패 화면(중복 구독·검증 실패 등)도 미리 만들어야 합니다. 도구별로 에러를 돌려주는 방법입니다.

# Python stub — 에러 던지기
def create_subscription(**kw):
    raise PaymentAPIError(409, "CONFLICT", "이미 활성 구독이 있습니다")  # → 409 화면 테스트

// MSW — 에러 상태코드로 응답
http.post('*/api/v1/subscriptions', () =>
  HttpResponse.json({ error: { code: 'CONFLICT', message: '이미 구독 중' } }, { status: 409 }))

// responses(Python) — 상태코드 지정
responses.add(responses.POST, f"{base}/subscriptions",
              json={"error": {"code": "VALIDATION_ERROR", "message": "형식 오류"}}, status=422)

각 코드의 응답 본문은 errors.json에 그대로 있으니 복사해 쓰면 됩니다.

9.4요청 본문 예시

POST 요청에 보낼 본문 형식입니다. 사전 개발 단계에선 auth_key는 아무 문자열로 둡니다.

// POST /api/v1/subscriptions  (구독 생성)
{ "external_user_id": "user-1001", "plan_id": "2222…2222",
  "auth_key": "dev-auth", "customer_key": "cust-1001", "trial": false }

9.6모든 화면 상태 만들어 보기

구독 상세 화면은 상태에 따라 보여줄 내용·가능한 버튼이 달라집니다. 아래 덤프로 모든 상태를 빠짐없이 개발·검증하세요.

상태덤프 파일access_allowed화면에서 확인할 것
ACTIVEsubscription.get.jsontrue정상 이용·다음 결제일·카드
TRIALsubscription.trial.jsontrue체험 종료일 안내
PAST_DUEsubscription.past_due.jsontrue결제 실패 경고·재시도(retry_count)
CANCELEDsubscription.canceled.jsontrue해지 예약·만료일까지 이용 안내
SUSPENDEDsubscription.suspended.jsonfalse이용 차단 화면·재결제 유도
EXPIREDsubscription.expired.jsonfalse종료 화면·재구독 유도(card=null)
구독 없음응답 없이 404 (errors.json의 404)—"구독 없음" 빈 화면
내역 없음payments.empty.json—빈 결제 내역 화면

이용 허용은 상태를 일일이 따지지 말고 access_allowed 값 하나로 판단하면 됩니다. 위 표에서 false인 SUSPENDED·EXPIRED만 차단 화면, 나머지는 이용 가능 화면으로 처리하면 끝입니다.

에러 분기도 같은 방식으로 미리 검증합니다 — 409(중복 구독), 422(체험 미지원/형식 오류), 402(결제 실패), 404(없음), 429(한도), 503(서버 중단). 각 코드 응답은 errors.json에 있습니다.

개발 완료 체크리스트 (이만큼 만들면 끝)

  • ☐ 요금제 화면 — GET /plans로 요금제·첫 결제액·정기 결제액·체험/자동결제 배지 표시
  • ☐ 구독 시작 — POST /subscriptions 호출 후 결과 처리(성공/실패)
  • ☐ 접근 제어 — access_allowed로 기능 잠금/해제
  • ☐ 내 구독 화면 6종 — ACTIVE·TRIAL·PAST_DUE·CANCELED·SUSPENDED·EXPIRED + "구독 없음(404)"
  • ☐ 결제 내역 — 혼합 목록과 빈 내역(payments.empty.json) 둘 다
  • ☐ 단건 결제·취소 — 결제 결과 + 취소 후 수수료·환불 표시
  • ☐ 에러 분기 — 409·422·402·404·429·503 각 메시지/화면
  • ☐ 전환 준비 — HTTP 호출부를 한 곳에 모아 두기(나중에 실서버로 교체 쉽게)

9.8실제 서버로 전환 체크리스트

  • 덤프 로드(stub/MSW)를 실제 HTTP 호출로 교체 — 엔드포인트·메서드·응답 필드 동일하므로 화면·로직은 그대로.
  • 베이스 URL을 실제 결제 서버로 변경.
  • 요청에 HMAC 서명 헤더 4종 적용(7.3~6.4).
  • 프런트에 실제 토스 클라이언트 키 + 카드 등록창(requestBillingAuth) 연동(7.11).
  • 호출 서버 공인 IP를 허용 IP에 등록 요청(누락 시 403).
  • 금액은 서버가 계산하므로(덤프와 다를 수 있음) 표시 금액을 서버 응답 기준으로 확인.
  • 첫 결제 실패 시 402·구독 미생성 동작 반영(2.7).
