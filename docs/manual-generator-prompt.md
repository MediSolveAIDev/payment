# 매뉴얼 생성 프롬프트 (다른 서비스 재사용용)

> payment_system의 `docs/user_manual/` 매뉴얼 구조를 다른 서비스 레포에서 그대로 만들 때
> 클로드에게 통째로 붙여넣는 프롬프트. `<서비스명>`·`<...>` 부분만 채워서 사용.

---

## 붙여넣을 프롬프트

이 레포(<서비스명>)에 정적 HTML **매뉴얼 사이트**를 만들어줘. 구조·동작은 아래 사양을 그대로 따른다.

### 0. 엔진 + 디자인 이식 (가장 먼저 — 필수)
**디자인(색·폰트·레이아웃·콜아웃·코드배지 등 모든 비주얼)은 아래 `manual.css`/`manual.js`에 들어있다. 이 두 파일을 복사해야 디자인이 동일하게 나온다.** 이 프롬프트의 글만으로는 비주얼이 재현되지 않으니, 반드시 참조 레포(payment_system)의 아래 3개를 그대로 복사해 빌드 엔진으로 쓰고 `DOCS` 목록만 이 서비스에 맞게 교체해라.
- `docs/user_manual/build.py`            — 변환 로직
- `docs/user_manual/assets/manual.css`   — **디자인 시스템(컬러 토큰·Pretendard 폰트·사이드바·콜아웃·배지·그림자 등 ~200줄). 디자인의 본체.**
- `docs/user_manual/assets/manual.js`    — 검색·목차 스크롤

> 3파일을 복사할 수 없는 환경이면 1~4 사양대로 새로 구현하되, **그 경우 비주얼은 동일하지 않다**(기능만 유사). 디자인 일치가 필요하면 `manual.css`를 반드시 함께 가져와라.

### 1. 산출물 구조
```
docs/user_manual/
  NN-slug.md          # 소스(마크다운). 두 자리 번호 + 케밥 slug. 예: 00-overview.md
  NN-slug.html        # build.py가 생성(직접 편집 금지)
  index.html          # 그룹 카드 목록 표지(생성물)
  build.py            # 마크다운 → HTML 변환기
  assets/
    manual.css        # 스타일(좌측네비·우측목차·콜아웃·코드배지·검색·.flow 다이어그램)
    manual.js         # 검색·목차 스크롤 등
    img/              # 스크린샷 등(있으면)
```

### 2. build.py 사양
- 상단에 `DOCS = [("NN-slug.md", "<그룹명>"), ...]` 리스트로 **사이드바 순서·그룹**을 정의.
  그룹 예: `"사용자 매뉴얼"`(📘 처음 쓰는 사람용 전체 흐름·화면) / `"개발자 매뉴얼"`(🛠️ 설치·배포·API·코드 흐름). 서비스에 맞게 그룹명/아이콘 조정 가능.
- 변환 기능(이식하면 그대로 동작):
  - 좌측 그룹별 네비, 우측 페이지 내 목차("이 페이지 내용", h2/h3 자동 수집)
  - **콜아웃**: 마크다운 인용구(`>`)의 첫 단어로 분류 — `주의/경고/중요`→⚠️, `참고/팁/예)`→💡, `쉽게 말하면/비유하면`→🍎, `관련 문서/함께 보기`→🔗, `대상 독자/권한`→👤, 기타→💬
  - **코드위치 배지**: `` `app/main.py:78` `` 처럼 `파일:라인` 패턴의 인라인 코드를 자동으로 클릭형 배지(`code.loc`)로
  - **문서 간 링크**: 본문의 `[글](NN-slug.md)` 를 빌드시 `.html`로 자동 치환
  - **표지/요약**: 각 문서의 첫 `# N. 제목`을 타이틀로, "한 줄 요약" 섹션(또는 첫 문단)을 카드 요약으로 사용
  - **검색창**, **`.flow` 프로세스 다이어그램** 지원
- 실행: `uv run --with markdown python docs/user_manual/build.py` → 전체 문서 + index.html 재생성.

### 3. 마크다운 작성 규칙
- 각 문서는 `# N. 제목`으로 시작하고, 바로 아래 **"한 줄 요약"** 한 문장을 둔다.
- 화면·기능 설명은 실제 코드 기준으로, 근거가 되는 위치를 `` `파일:라인` `` 으로 인라인 표기(배지화됨).
- 강조/주의/팁은 인용구(`>`)로 작성해 콜아웃이 자동 적용되게 한다.
- 한국어로 작성.

### 4. 내용 채우기 (이 서비스 분석)
이 레포를 분석해서 다음을 문서로 구성해라(서비스에 없는 항목은 생략):
- **사용자 매뉴얼 그룹**: 개요/전체 프로세스, (관리자 콘솔이 있으면) 화면별 사용법, 대시보드, 상태·용어 레퍼런스
- **개발자 매뉴얼 그룹**: 설치·환경설정·배포(docker/nginx 등), 외부 연동 API, 기능별 코드 흐름, (있으면)샘플/연동 서비스
- 번호(`NN`)와 그룹은 위 구조를 참고해 이 서비스 실제 구성에 맞게 재배치.

### 5. 마무리
- `build.py` 실행해서 `index.html` + 전 페이지가 정상 생성되는지 확인하고, 깨진 링크/빈 페이지가 없는지 점검.
- 이 레포의 작업 규칙(주석 필수, 작업내역 워크로그 등)을 따른다.

## 전달물 (사용할 사람에게 넘겨야 할 것)

이 프롬프트만 줘서는 디자인이 재현되지 않는다. 아래를 **한 묶음**으로 전달한다.

1. **이 프롬프트** — `docs/manual-generator-prompt.md` (이 파일)
2. **엔진·디자인 3파일** (디자인 일치의 핵심 — 빠지면 비주얼이 달라짐)
   - `docs/user_manual/build.py`
   - `docs/user_manual/assets/manual.css`
   - `docs/user_manual/assets/manual.js`
3. **참고 예시 1개** (선택, 권장) — 완성된 마크다운 문서 하나(예: `docs/user_manual/00-overview.md`).
   "한 줄 요약·콜아웃·`파일:라인` 배지"를 실제로 어떻게 쓰는지 보여주는 샘플.

받는 사람 사전 준비물:
- **Python 3** + **uv**(권장). 빌드는 `uv run --with markdown python docs/user_manual/build.py` 로 실행 → `markdown` 패키지를 uv가 자동 설치. (uv가 없으면 `pip install markdown` 후 `python build.py`)
- 폰트(Pretendard)는 CSS에서 시스템 폰트로 자동 폴백되므로 별도 설치 불필요. 완전히 동일한 폰트를 원하면 Pretendard 웹폰트를 추가.

전달 방법 예:
- 같은 모노레포면 경로 그대로 복사. 별도 레포면 위 1~3을 압축해 전달하거나, payment_system 레포 접근 권한을 주고 "0번대로 3파일 복사"를 지시.

## 사용 팁
- 처음엔 문서 1~2개만 만들어 빌드가 도는지 확인한 뒤 나머지를 채우는 게 안전하다.
- 디자인을 100% 동일하게 하려면 0번(엔진 3파일 복사)을 꼭 먼저 시키고, 디자인 변형을 원하면 `manual.css`만 손대게 한다.
