# PostgreSQL DB 백업 프로그램 — 작업 계획 (plan.md)

> 결제/구독 시스템(`payment_system`)의 PostgreSQL 데이터베이스를 **6시간마다 자동 백업**하는
> 프로그램 개발 계획. 작성일: 2026-06-18.
>
> 🖥️ **실행 위치: PostgreSQL이 설치된 바로 그 서버에서 로컬로 실행**한다. DB와 백업 프로그램이
> 같은 호스트에 있으므로 네트워크를 거치지 않고 로컬 디스크에 백업 파일을 만든다.
> (외부 관리형 DB가 아니라 **자체 설치형 PostgreSQL** 기준)
>
> ⚠️ **전송(다른 서버로 보내기)은 이번 범위에서 제외**한다. 백업 파일을 "안전하게 생성·검증·보관"
> 하는 데까지를 목표로 하고, 원격 전송은 [§10 후속 단계](#10-후속-단계전송)에 설계만 남겨 둔다.

## 1. 목표

- 운영 PostgreSQL DB를 **6시간 주기**로 일관성 있게 덤프한다.
- 덤프 파일을 **압축·무결성 검증·메타데이터 기록**한다.
- **보존 정책**에 따라 오래된 백업을 자동 정리(prune)한다.
- 실패 시 **로그·알림**으로 즉시 인지할 수 있게 한다.
- 무엇보다 **복구(restore)가 실제로 되는지** 검증한다(복구 안 되는 백업은 백업이 아니다).

## 2. 대상 환경

| 항목 | 값 |
|------|----|
| DBMS | PostgreSQL **16** (자체 설치형 — 백업 프로그램과 **같은 서버**에 설치됨) |
| 실행 위치 | **DB 서버 로컬**. pg_dump가 같은 호스트에서 로컬 접속(소켓/`localhost`) |
| 접속 정보 | 로컬 접속 + **백업 전용 읽기 계정** 권장(또는 peer 인증 / `.pgpass` — §8) |
| 기존 백업 흔적 | `backups/payment_backup_pre_reset.sql` (수동 `pg_dump` 산출물) |
| 런타임 생태계 | Python 3.13 · `uv` (저장소 일관성) |
| 핵심 함의 | DB와 같은 호스트 → ① `pg_dump` 버전이 **자동으로 서버와 일치** ② 네트워크 불필요 ③ **cron이 가장 자연스러운 스케줄러** |

## 3. 범위

**In scope**
- 백업 실행 CLI(단발 1회 실행 = 1백업), 6시간 스케줄링, 압축, 무결성 검증, 보존 정리, 로깅, 실패 알림, 복구 CLI·복구 리허설.

**Out of scope (이번엔 안 함)**
- 다른 서버/클라우드로의 **원격 전송**(설계만 §10에 남김).
- PITR(Point-In-Time Recovery)·WAL 아카이빙 — 관리형 Azure DB가 자체 제공하므로 이 도구는 **논리 백업(pg_dump)** 만 담당.

## 4. 추천 언어 · 프레임워크 ⭐

### 결론
- **언어: Python 3.13** — 기존 저장소와 동일 생태계(테스트·로깅·보존 로직·알림을 한 곳에서 깔끔히).
- **백업 엔진: `pg_dump` / `pg_restore`** (PostgreSQL 공식 클라이언트) — DB 백업은 직접 구현하지 않고 검증된 표준 도구를 호출한다.
- **CLI: Typer** — FastAPI와 같은 계열(타입 힌트 기반), 하위 명령(`backup`/`restore`/`prune`/`list`) 구성에 적합.
- **설정: pydantic-settings + `.env`** — 기존 `app/core/config.py` 패턴과 동일하게 환경변수 로딩.
- **패키징/실행: `uv`** — 저장소와 동일 툴체인.
- **스케줄링: DB 서버의 cron(또는 systemd timer)** 으로 단발 CLI를 6시간마다 호출. DB와 같은 호스트에서 도는 단발 작업이라 cron이 가장 단순·견고하다(컨테이너화는 불필요 — 자체 설치 PostgreSQL 옆에서 직접 실행).

### 왜 이 조합인가
- **pg_dump는 타협 불가**: 파이썬에서 테이블을 직접 읽어 백업하면 트랜잭션 일관성·대용량·복원 호환을 보장 못 한다. `pg_dump`는 단일 트랜잭션 스냅샷으로 **일관된** 백업을 만든다.
- **스케줄은 OS에 위임이 더 견고**: 6시간마다 "상주 프로세스"를 띄우면 그 프로세스가 죽으면 백업도 멈춘다. cron/systemd timer는 매번 새 프로세스를 띄우므로 **단발 실패가 다음 주기에 영향 없음**(멱등·격리).

### 대안 비교
| 방식 | 장점 | 단점 | 평가 |
|------|------|------|------|
| **Python + pg_dump + cron** ⭐ | 저장소 일관성, 테스트·보존·알림 용이, 멱등 | 의존성 약간 | **권장** |
| Bash + pg_dump + cron | 초경량, 의존성 거의 없음 | 보존/검증/알림 로직이 지저분, 테스트 어려움 | **DB 서버 로컬 단발 작업이라 충분히 합리적**(요구가 단순하면) |
| Python + APScheduler(상주) | 자기완결 | 프로세스 죽으면 정지, 중복 실행 주의 | 로컬 cron이 있으니 불필요 |
| pgBackRest / barman (전용 솔루션) | 증분·PITR·병렬 등 강력 | 러닝커브·과한 기능 | 로컬 물리백업·PITR이 꼭 필요해지면 그때 검토 |

> 참고: DB와 같은 호스트라 파일시스템 접근이 되므로 `pg_basebackup`/PITR 같은 **물리 백업**도 가능하지만,
> 이번 목표(6시간 주기 논리 백업 파일 생성)에는 **`pg_dump`가 가장 단순·적합**하다.

## 5. 아키텍처 / 동작 흐름

```
[cron / systemd timer]  ──6시간마다──▶  python -m db_backup backup
                                              │
                                              ▼
   1) pg_dump -Fc  (custom 포맷, 압축 내장, 단일 트랜잭션 스냅샷)
                                              │
                                              ▼
   2) 무결성 검증  pg_restore --list <파일>   (목록이 정상 출력되면 OK)
                                              │
                                              ▼
   3) 메타데이터 기록  (크기/소요시간/행수·체크섬 → manifest.json)
                                              │
                                              ▼
   4) 보존 정리  prune  (정책보다 오래된 백업 삭제)
                                              │
                                              ▼
   5) 결과 로그 + 실패 시 알림(메일/슬랙)
```

- 한 번의 실행 = **하나의 멱등한 백업 작업**. 스케줄러는 이 명령을 6시간마다 부를 뿐.
- 파일명 규칙(예): `payment_YYYYMMDD-HHMM_<git_or_db_ver>.dump` (+ `.sha256`).

## 6. 백업 전략 상세

- **포맷**: `pg_dump -Fc`(custom). 압축 내장 + `pg_restore`로 **선택적/병렬 복원** 가능. 평문 `.sql`보다 우월.
  - 이식성 위해 복원 시 `--no-owner --no-privileges` 옵션 고려(대상 롤이 다를 때).
- **일관성**: pg_dump는 기본적으로 일관된 스냅샷(단일 트랜잭션)을 뜬다 — 별도 락 불필요.
- **글로벌 객체**: 단일 DB라 `pg_dump`로 충분. 롤/권한까지 필요하면 `pg_dumpall --globals-only` 별도 1일 1회(선택).
- **무결성 검증**: 덤프 직후 `pg_restore --list`로 카탈로그가 읽히는지 확인 + `sha256` 체크섬 저장.
- **암호화(선택)**: 백업에 카드 빌링키 암호문 등이 포함되므로, 저장 디스크가 신뢰 경계 밖이면 `age`/`gpg`로 at-rest 암호화 검토(§8).
- **pg_dump 버전**: 클라이언트 `pg_dump` 메이저 버전 ≥ 서버 버전이어야 안전. 컨테이너 베이스를 **PostgreSQL 16 클라이언트**로 맞춘다.

## 7. 보존 정책 (Retention) — 추천값

> 6시간 주기 = 하루 4개 생성. 비용 대비 복구 시점 커버리지를 고려한 **추천 기본값**:

| 단계 | 보존 | 개수(대략) | 목적 |
|------|------|-----------|------|
| 시간별(6h) | 최근 **2일** | 8개 | 최근 장애 즉시 롤백 |
| 일별(매일 첫 백업 승격) | 최근 **14일** | 14개 | 며칠 전 상태 복구 |
| 주별(매주 첫 백업 승격) | 최근 **8주** | 8개 | 장기 회귀 추적 |

- 위는 권장값이며 `.env`로 조정 가능(`KEEP_HOURLY`/`KEEP_DAILY`/`KEEP_WEEKLY`).
- 더 단순하게 가려면 "최근 28개만 유지(=7일치)" 롤링도 가능 — **미결정 사항(§11)** 에서 확정.

## 8. 설정 (환경변수, `.env`)

| 변수 | 예시/기본 | 설명 |
|------|-----------|------|
| `DATABASE_URL` | `postgresql://backup_ro@/payment` (로컬 소켓) 또는 `...@localhost:5432/payment` | 백업 대상. DB가 같은 호스트라 **로컬 소켓/`localhost`** 접속. **읽기 전용 백업 계정** 권장 |
| `BACKUP_DIR` | `/var/backups/payment` | 백업 파일 저장 경로(로컬 디스크 — DB 데이터와 **다른 디스크/파티션** 권장) |
| `KEEP_HOURLY/DAILY/WEEKLY` | `8 / 14 / 8` | 보존 정책(§7) |
| `BACKUP_ENCRYPT` | `false` | at-rest 암호화 사용 여부 |
| `ALERT_WEBHOOK` / `GMAIL_*` | (빈값) | 실패 알림 채널(저장소 메일 설정 재사용 가능) |
| `PG_DUMP_PATH` | `pg_dump` | 클라이언트 경로(버전 고정 시) |

- **백업 전용 계정**: 앱 계정과 분리해 최소 권한(읽기) 부여 → 자격증명 유출 시 피해 최소화. 로컬이면 **peer/ident 인증**이나 `.pgpass`로 비밀번호 없이도 운용 가능.
- 비밀값은 Git 미추적(`.env`), 예시는 `.env.example`로 제공.
- **저장 위치 분리**: 백업 파일은 DB 데이터 디스크와 **다른 디스크**에 둔다 — 데이터 디스크가 깨지면 백업도 같이 잃는 것을 막기 위함(같은 서버라 더 중요).

## 10. 작업 계획 (Phase)

- [ ] **Phase 0 — 사전 준비/결정**
  - 백업 전용 DB 계정(읽기) 생성, 저장 경로·볼륨 확보, §11 미결정 사항 확정.
- [ ] **Phase 1 — 핵심 백업 CLI** `backup`
  - `pg_dump -Fc` 실행 → 파일명 규칙 → 저장. `--dry-run` 지원.
- [ ] **Phase 2 — 검증 + 메타데이터**
  - `pg_restore --list` 무결성 확인, `sha256`·`manifest.json` 기록.
- [ ] **Phase 3 — 보존 정리** `prune`
  - GFS 정책 구현 + 단위 테스트(시간 모킹).
- [ ] **Phase 4 — 스케줄링(DB 서버 로컬)**
  - DB 서버에 cron 6시간 등록(예: `0 */6 * * *`) 또는 systemd timer. 로그 로테이션 설정.
- [ ] **Phase 5 — 로깅/실패 알림**
  - 성공/실패 구조적 로그, 실패 시 메일/웹훅. 헬스(마지막 성공 시각) 노출.
- [ ] **Phase 6 — 복구 절차 + 리허설** ⭐
  - `restore` CLI, README에 단계별 복구 가이드, **주기적 복구 테스트**(임시 DB로 자동 왕복).
- [ ] **Phase 7 (후속) — 다른 서버로 전송**
  - 아래 §10 후속 단계 참고. 이번 범위 아님.

### 10-후속 단계(전송)
> 이번엔 구현하지 않지만, 나중에 붙일 자리만 명시:
> - 인터페이스를 `Storage`(local) ↔ `RemoteTransport`로 분리해 두면, 추후 `rsync/scp`·`Azure Blob`·`S3`
>   백엔드를 **플러그인처럼** 추가 가능.
> - 전송 성공 후 원격 체크섬 재검증 → 로컬 보존 정책 적용 순서 권장.

## 11. 미결정 사항 (확정 필요)

> ✅ **실행 위치 확정** — PostgreSQL이 설치된 서버에서 로컬 cron으로 실행(이 항목은 결정됨).
> ✅ **전송 제외 확정** — 이번 범위 아님.

1. **보존 정책** — §7 GFS(추천) vs 단순 N개 롤링.
2. **알림 채널** — 메일(저장소 GMAIL 설정 재사용) vs 슬랙/웹훅 vs 없음(로그만).
3. **at-rest 암호화** — 백업 디스크 접근 통제가 약하면 필요. *(빌링키 암호문 포함 주의)*
4. **인증 방식** — 백업 전용 계정 + `.pgpass` vs peer 인증.
5. **백업 저장 경로/디스크** — `BACKUP_DIR`와 DB 데이터 디스크 분리 가능 여부.

## 12. 보안 · 운영 주의

- 백업엔 **결제·빌링키 암호문**이 들어간다 → 저장 디스크 권한 최소화, 필요 시 암호화, 접근 감사.
- `ENCRYPTION_KEY`(앱)와 백업은 **별개로 안전 보관** — 키 없이는 빌링키 복호화 불가하므로 키 백업도 병행.
- 백업 파일·`.env`는 Git 미추적. 컨테이너는 비루트 실행.

## 13. 성공 기준 (Definition of Done)

- 6시간마다 백업이 생성되고, 매 백업이 `pg_restore --list`로 검증된다.
- 보존 정책이 자동 적용되어 디스크가 무한 증가하지 않는다.
- **임시 DB로 복구 왕복 테스트가 통과**한다(복구 가능성 입증).
- 실패 시 알림이 도착하고, 마지막 성공 시각을 확인할 수 있다.
